# 项目初心

如果只从功能上看，这个项目当然可以被描述成一个"基于 B 站视频字幕的 RAG 问答系统"。但对我来说，它真正的起点并不是"学校要做一个课程设计，所以我选了这个题目"——更准确地说，它是几条线慢慢汇到一起之后，自然长出来的一个项目。

## 1. 最早吸引我的，其实不是 RAG，而是结构化数据和自动化处理

我一直对结构化数据、自动化处理、RPA、爬虫这一类东西很感兴趣。这种兴趣不是停留在"知道有这个方向"，而是会让我天然去关注那些能把杂乱信息整理成结构、再进一步做分析和处理的事情。

也是因为这个原因，我之前做过一些和 B 站相关的尝试——比如爬取用户自己的收藏夹，然后分析这个用户大概是一个什么样的人。那个项目最后并没有完整落地，但它让我实际接触到了 B 站的鉴权、接口抓取，以及这类平台数据在真实环境里的复杂性。

从那时起，我对 B 站就不再仅仅停留在"用户"的视角，而是多了一层"数据和接口"的视角。

## 2. 我看中了 Bilibili Reader

当我在学习 spring-ai-alibaba 框架、阅读它的文档时，我一眼注意到了藏在十几个 Reader 里面的 Bilibili Reader。这并不是偶然——因为我本来就对 B 站抓取和鉴权比较熟悉，也知道这类事情真正难的地方从来不是"有没有这个名字"，而是它到底能不能在真实环境里把数据拿回来。

结果一看，问题很明显：它一开始并没有凭证支持，接口本身也存在问题，导致根本无法稳定获取字幕。对一个视频字幕 Reader 来说，这不是小瑕疵，而是根本性问题。

所以我对它的态度，不再是"用一下看看"，而是变成了"我想把它修好"。我开始重构它、补上凭证支持、修接口错误，并尝试把这部分能力往上游开源项目里提交 PR。这一步对我来说很关键——从这里开始，我和这个组件之间的关系就变了：我不再只是一个使用者，而是开始变成一个修复者、适配者、贡献者。

## 3. 课程设计，给了这条线一个正式落地的机会

后来学校给出了一批课程设计的题目，我几乎是一眼就相中了"基于 AI 的问答系统"这个方向。原因并不是它单纯"和大模型有关"，而是它刚好接上了我前面已经在做的一条线：B 站 Reader 我已经关注过、修过，也知道它真正的价值不在于"这个组件写出来了"，而在于它能不能被放进一个真实场景里跑起来。

于是我想到：那就干脆把它放进一个完整项目里，真正跑通一次。

所以这个项目从一开始想验证的，就不只是“能不能做一个聊天页面”或“能不能把大模型接起来”，而是一个我真正关心、也亲手修过的组件，能不能在完整链路和真实场景里站住。

## 4. 我真正想验证的，不只是"能不能做出来"

如果只从表面看，这个项目完成的是一个典型的 RAG 闭环：用户导入 B 站视频、获取字幕、进行清洗和切分、写入向量库，再通过检索增强问答把内容重新组织出来。但对我来说，它的意义比这更具体。

### 4.1 验证组件能力

我真正关心的第一件事，是修过的 Bilibili Reader 能不能在真实项目里真正成立。很多东西只做成组件，并不代表它已经有价值——只有把它放进真实的导入、索引、检索、问答流程里，才知道它到底是"看起来能用"，还是"真的能用"。这个项目首先就是这样一个验证场。

### 4.2 验证完整的 RAG 链路

第二件事，是我想亲手跑通一条从视频内容到问答结果的完整链路。我想验证的不只是某一段代码，而是这条完整流程是否真的闭合：**字幕抓取 → 文本处理 → 向量化 → 检索 → 问答**。只有这条链路真正跑起来，前面的组件修复才算真正落了地。

### 4.3 验证工程能力

第三件事同样重要：这个项目也是我第一次比较认真地把 Git、协作、分支、CI/CD 这些工程能力放进一个真实项目里练。所以它不只是一个"能跑起来"的系统，也是一块真正用来练工程习惯的场地。

## 5. DashVector 这条线也很有意思：做是做出来了，但它确实很别扭

除了 Bilibili Reader 之外，DashVector 也是这个项目里非常重要的一条线。

一开始我的想法很直接：既然上游没有 DashVector 的组件，那我就自己做一个，做完之后再考虑要不要开源。结果真正做进去之后，我很快就发现，问题不是"能不能做"，而是"做出来之后够不够自然"。

最后它当然是做出来了，也确实跑通了——项目里的向量检索链路就是靠它打通的。但整个适配过程里，我一直有一种非常明显的别扭感。这个别扭感不是来自某一个 bug，而是来自它底层那套东西和 Spring AI 的抽象天然就不太像一路人。比如：

- Spring AI 里 `Document.text` 是核心正文，但在适配时不得不额外塞进一个 `content` 字段再转回来
- Filter 不是天然对齐的，只能自己写转换器把 Spring AI 的表达式编译成 DashVector 的过滤语法
- Score / Similarity 的语义也不完全一致，需要自己重新解释
- 很多地方并不是"优雅接入"，而更像是在"想办法把它接上"

所以最后我的感受并不是"这玩意做不出来"，而是：**能适配，也能跑通，但抽象并不成立，至少没有自然成立。**

这也是为什么我最后没有把它轻易当成成熟组件往上游推进。不是因为它没用，而是因为我自己知道，它还没有达到那种"我愿意很安心拿出去展示"的程度。

不过反过来说，如果以后也有读者打算自己接 DashVector，那我在这个项目里留下来的代码和踩坑过程，也许正好能给你一点参考：至少你会更早知道，问题不一定出在你不会写，而可能就是它底层那套东西和 Spring AI 的抽象本来就不太对路。

## 6. 这个项目也是我第一次认真练习 Git、协作和工程规范的地方

这个项目还有一层很现实的意义：它不只是让我验证了组件和功能，也让我第一次比较认真地去练习工程化协作。

如果读者去翻这个仓库，会看到多人协作的痕迹、分支的痕迹、PR 合并的痕迹，也能看到 CI/CD 的配置和自动化流程。它不是一个完全刻意打磨成"展示品"的仓库，而是一个真实项目留下来的过程痕迹。

但对我来说，这里面最重要的，不只是"提交信息是不是写了 `feat:`、`fix:`"，而是我开始真正意识到 Git 规范不只是写 commit message，而是：

- 什么时候应该为独立功能开分支
- 什么时候该合并
- 怎样减少主线污染
- 怎样让功能边界和 Git 边界尽量一致
- 怎样在多人协作时保持更清晰的节奏

在给上游开源项目提交 PR 之后，我才慢慢开始真正重视这些事情。以前我也会用 Git，但那个阶段的"会用"和后来理解到的"工程化地使用 Git"，其实不是一回事。这个项目刚好就成为了这些能力第一次被我放进真实项目里认真练习、去验证、去留下痕迹的地方。

## 7. 以前我只是知道 CI/CD 存在，这个项目让我第一次真正把它用起来

在做这个项目以前，我当然知道开源项目会跑 CI/CD，也知道很多成熟项目会在 push、PR、发布这些节点上自动做很多事。但那种"知道"，更多只是旁观者视角。

这个项目让我第一次真正把这些东西用在自己的项目里。前端这边，我用上了 Cloudflare Pages 的自动部署能力——当主分支有变动时，Cloudflare 会自动拉取最新代码并完成部署，不需要我手动操作任何东西；后端这边，也做了打包和上传服务器的流程。对今天的我来说，这未必是什么特别大的事情，但对当时的我来说，这代表的是一种很真实的变化：很多以前只是"知道"的东西，终于开始变成"我自己真的用过、而且放进项目里跑过"的东西。

## 8. 所以，这个项目真正想证明的，不只是"我做了一个 RAG 应用"

如果只看功能，它当然是一个 B 站视频字幕问答系统。但如果只这么理解它，我会觉得太浅了。

因为它真正想承载的，其实是这些事情第一次被我串到一起：

- 对结构化数据和自动化处理的长期兴趣
- 对 B 站接口和抓取问题的实际经验
- 对上游组件的修复与重构
- 把组件放进真实场景里做完整验证
- 对 Git、分支、协作和工程规范的认真练习
- 对 CI/CD 从"知道"到"真正上手"的跨越
- 对"适配成功"与"抽象成立"之间区别的第一次深刻认识

所以这个项目不是简单意义上的"课设作品"，也不只是"一个能跑的 RAG demo"——它更像是我把兴趣、组件、应用、开源、工程规范和实战验证第一次真正连起来的地方。

## 9. 如果只用一句话总结

这个项目诞生的初心，不只是为了做一个 B 站 RAG 应用，而是为了把我真正关心、亲手修过、并愿意继续打磨的能力，放进一个完整而真实的工程场景里，验证它到底能不能成立；同时也借这个过程，第一次认真练习我对 Git、协作、工程规范和 CI/CD 的理解与使用。
