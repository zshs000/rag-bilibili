# 项目初心：不止于 B 站 RAG，更是一次全链路工程化验证
单从功能定义，这是一个**基于 B 站视频字幕的 RAG 问答系统**；但于我而言，它绝非一次应付课程设计的选题，而是我长期的技术兴趣、实战积累与工程探索交织融合后，自然生长出的完整项目。

## 1. 初心起点：钟情于结构化数据与自动化处理
我始终对结构化数据、自动化处理、RPA、网络爬虫等方向抱有浓厚兴趣。

这份兴趣不止于认知层面，更驱动我主动探索如何将无序信息规整为结构化数据，并基于此完成分析与落地应用。

此前，我曾尝试开发 B 站收藏夹用户画像分析项目，虽未完整落地，却让我深度接触了 B 站鉴权机制、接口抓取，也直面了平台数据在真实生产环境中的复杂性。

自此，我对 B 站的认知，从普通用户视角，升级为数据与接口的开发者视角。

## 2. 核心契机：深耕并修复 Bilibili Reader 组件
在学习 spring-ai-alibaba 框架时，我第一时间关注到了框架内置的 Bilibili Reader 组件。

这并非偶然——凭借此前 B 站数据抓取的实战经验，我深知这类组件的核心难点，不在于功能命名，而在于真实环境下的数据稳定获取。

实测后我发现核心问题：组件缺失凭证支持、接口逻辑存在缺陷，导致字幕无法稳定抓取，这对于视频字幕读取组件而言，是致命的底层问题。

于是我不再止步于使用者，而是着手**重构组件、补全凭证体系、修复接口漏洞**，并尝试向上游开源项目提交 PR。

这一步成为关键转折点：我从组件使用者，转变为修复者、适配者与开源贡献者。

## 3. 落地载体：课程设计，让技术积累正式闭环
学校发布课程设计选题时，我毫不犹豫选择了**基于 AI 的问答系统**方向。

选择的核心原因并非跟风大模型技术，而是它完美承接了我此前的技术积累——我已深耕并修复 Bilibili Reader，而我最想验证的，正是这个组件能否在完整业务场景中稳定落地。

最终我确定了方案：将该组件融入完整项目，跑通全流程业务闭环。

这个项目从立项之初，目标就不是简单搭建聊天界面、接入大模型，而是验证我亲手优化的核心组件，能否在真实链路中稳定运行。

## 4. 核心目标：不止于实现功能，更在于深度验证
表面上，项目完成了标准 RAG 业务闭环：用户导入 B 站视频、抓取字幕、文本清洗切分、写入向量库，最终通过检索增强生成实现智能问答。

但对我而言，其核心价值远不止于此。

### 4.1 验证核心组件的工程可用性
组件独立开发完成，不代表具备实际价值。

我首要验证的，是修复后的 Bilibili Reader 能否融入**导入、索引、检索、问答**全流程，区分「看似可用」与「真正工程可用」，这个项目正是组件能力的核心验证场。

### 4.2 验证 RAG 全链路的完整性
我希望亲手跑通**字幕抓取 → 文本处理 → 向量化 → 向量检索 → 智能问答**的完整链路。

唯有整条链路无缝闭环，此前的组件优化工作，才算真正落地生效。

### 4.3 验证个人工程化实战能力
这是我首次系统性将 Git 协作、分支管理、CI/CD 等工程化能力落地于实战项目。

项目不仅是可运行的应用系统，更是我打磨工程化开发习惯的实践阵地。

## 5. 技术思考：DashVector 适配，能落地但抽象不兼容
除 Bilibili Reader 外，DashVector 向量库适配是项目另一核心技术线，也让我形成了深刻的技术认知。

立项初期，我的规划很明确：上游缺失 DashVector 适配组件，便自主开发并规划开源。

但深入开发后我发现，难点不在于功能实现，而在于**架构抽象的兼容性**。

最终我完成了全量适配，成功打通项目向量检索链路，但整个开发过程始终存在明显的违和感。

这种违和感并非源于 Bug，而是其底层设计与 Spring AI 原生抽象体系存在天然割裂：
- Spring AI 以 `Document.text` 为核心文本载体，适配时需额外封装 `content` 字段做双向转换；
- 过滤表达式无法原生对齐，需自研转换器将 Spring AI 语法编译为 DashVector 专属语法；
- 相似度评分语义不统一，需手动重定义评分规则；
- 多数场景无法优雅接入，仅能通过兼容方案强行适配。

最终结论清晰：**功能可适配、链路可跑通，但架构抽象不成立，无法做到原生兼容**。这也是我未将其作为成熟组件向上游贡献的核心原因——它具备实用性，但未达到我心中可开源推广的高标准。

同时，这段踩坑经历也具备参考价值：若后续开发者尝试适配 DashVector，可参考本项目代码，提前规避架构兼容问题，明白适配难点并非开发能力，而是底层抽象的天然差异。

## 6. 能力成长：系统化实践 Git 协作与工程规范
这个项目的另一重价值，是让我完成了从「会用 Git」到「工程化使用 Git」的认知升级。

仓库中留存着多人协作、分支开发、PR 合并、CI/CD 自动化配置的完整痕迹。

它不是刻意包装的展示项目，而是真实开发过程的完整沉淀。

我真正理解的 Git 规范，远不止 `feat:`、`fix:` 等提交信息格式，而是整套工程化协作思维：
- 如何为独立功能拆分分支，保证功能边界清晰；
- 如何控制合并节奏，避免主线代码污染；
- 如何对齐功能边界与 Git 版本边界；
- 如何在多人协作中维持高效、清晰的开发节奏。

向上游开源项目提交 PR 的经历，让我彻底重视起这些工程规范。

而这个项目，正是我将这套认知落地实战、沉淀成果的核心载体。

## 7. 实战突破：从认知到落地，首次全量应用 CI/CD
此前我仅以旁观者视角了解 CI/CD，知晓成熟项目会在代码提交、PR 合并、版本发布等节点执行自动化流程，但从未亲身落地。

这个项目让我完成了从「认知」到「实战」的跨越：前端基于 Cloudflare Pages 实现自动部署，主分支代码更新后自动触发构建与发布；后端完成自动化打包、服务器上传全流程配置。

于当下而言，这是基础的工程化能力；但于当时的我而言，这是一次关键的成长——所有曾经只停留在概念中的技术，都真正落地为我亲手实践、稳定运行的能力。

## 8. 项目本质：串联兴趣与技术，完成个人成长闭环
若仅以功能定义，它只是一个 B 站字幕 RAG 问答系统；但这样的定义，完全低估了它的价值。

这个项目的核心意义，是将我多维度的技术积累与成长，第一次完整串联：
- 延续对结构化数据、自动化处理的长期兴趣；
- 沉淀 B 站接口开发、数据抓取的实战经验；
- 完成开源组件的修复、重构与上游贡献；
- 验证核心组件在真实业务场景的可用性；
- 系统化实践 Git 协作、分支管理与工程规范；
- 实现 CI/CD 从概念认知到落地应用的突破；
- 深刻理解「功能适配」与「架构抽象兼容」的核心区别。

它不是一份简单的课程设计作品，也不是一个演示用的 RAG Demo，而是我将**个人兴趣、组件开发、业务落地、开源协作、工程化思维**融为一体的成长里程碑。

## 9. 一句话总结
这个项目的初心，从来不是单纯开发一款 B 站 RAG 应用。

而是将我深耕打磨、亲手优化的核心技术能力，置于完整真实的工程场景中完成终极验证；同时借这个过程，系统性落地 Git 协作、工程规范与 CI/CD 能力，开启我的工程化与开源协作之路。
