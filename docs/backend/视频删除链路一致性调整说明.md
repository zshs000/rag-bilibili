# 视频删除链路一致性调整说明

## 背景

视频删除链路同时涉及 MySQL 和 DashVector。MySQL 操作可以由 Spring 事务管理，但 DashVector 删除属于外部系统调用，不会跟随 MySQL 事务一起提交或回滚。

这个问题和导入阶段的孤儿向量类似，都是跨资源一致性问题。但两者的失败方向不同：导入阶段主要担心“向量已写入，但数据库未完成落库”；删除阶段当前主要担心“向量已删除，但数据库删除未完成”。

## 当前删除顺序

当前 `VideoServiceImpl#deleteVideo()` 的顺序是：

1. 查询 `vector_mapping`，拿到待删除的向量 ID。
2. 调用 `dashVectorStore.delete(vectorIds)` 删除 DashVector 向量。
3. 删除 MySQL 中的 message、session、vector_mapping、chunk、video。

方法虽然标记了 `@Transactional`，但事务只覆盖第 3 步中的 MySQL 操作。第 2 步一旦成功，后续 MySQL 回滚也无法恢复 DashVector 中已经删除的向量。

## 风险

如果 DashVector 删除成功，但后续 MySQL 删除失败，最终状态会变成：

- MySQL 中视频记录仍然存在。
- DashVector 中对应向量已经丢失。

这不是高概率故障，但失败后的状态对用户更明显：视频仍在列表里，单视频问答也仍能找到 video 记录，但检索不到对应内容。

## 调整方向

删除链路更合理的顺序是：

1. 在 MySQL 事务中校验视频归属，并查询待删除的 `vectorIds`。
2. 在同一个 MySQL 事务中删除 message、session、vector_mapping、chunk、video。
3. MySQL 事务提交成功后，再调用 `dashVectorStore.delete(vectorIds)`。

这样即使 DashVector 删除失败，也只是留下孤儿向量。单视频场景下 video 已经不存在，不会继续进入该视频的检索链路；全部视频场景下残留向量仍限制在当前用户范围内，影响相对可控。

当前阶段的处理语义是：MySQL 删除成功后，接口即认为视频删除完成；后续 DashVector 删除失败只记录错误日志，不再向用户返回失败。用户主动重试删除不会清理这批残留向量，因为视频记录已经不存在。

## 必要性

这次调整不是为了引入完整的 outbox 或分布式事务，也不是把低概率问题扩大化处理。核心目标是让删除链路的失败方向符合项目已有的事实源判断：

- MySQL 是业务状态的权威来源。
- DashVector 是可补偿的外部索引。
- 优先避免“业务记录仍存在，但索引已经丢失”的状态。

后续如果需要更强的一致性，可以再考虑补偿任务或 outbox 表；当前阶段先调整删除顺序即可。
