# 字幕清洗多行窗口设计与实现说明

## 1. 背景与问题定义

本项目的视频导入链路里，字幕会在写入向量库前先经过“清洗 -> 切片 -> 向量化 -> 入库”的流程。

其中，字幕清洗的目标不是做通用文本纠错，而是尽量在切片前移除明显的广告和低信息噪声，避免这些内容进入向量库后污染检索结果。

本次优化之前，字幕清洗的主要问题是：

1. 自动字幕经常把一条广告语切成两行甚至三行。
2. 原有逻辑按“单行”判断是否删除，跨行广告会漏判。
3. 一旦漏判，这些广告会进入后续 chunk，影响 embedding 语义和检索质量。
4. 更关键的是，广告一旦进入向量库，就会变成 RAG 的候选知识片段；用户在提问相关关键词时，系统可能把广告字幕召回，再由大模型把广告内容当成视频知识的一部分回答出来。

典型漏判场景如下：

```text
感谢转转二手
对本期视频的支持
```

```text
点击下方
链接领取优惠
```

如果仍然只按单行判断，这类广告语很容易保留下来。

## 2. 原有实现与局限

原有实现位于：

- `rag-bilibili-server/src/main/java/com/example/ragbilibili/transformer/SubtitleCleaningTransformer.java`
- `rag-bilibili-server/src/main/java/com/example/ragbilibili/service/impl/VideoServiceImpl.java`

原有实现的处理方式是：

1. 读取 `Transcript:` 后的正文。
2. 按换行切成若干字幕 segment。
3. 对每个 segment 做标准化。
4. 对每个 segment 单独应用删除规则。
5. 删除相邻重复字幕。
6. 将清洗结果交给 `TokenTextSplitter` 做切片。

这个方案有两个优点：

- 实现简单，可解释性强。
- 保留了原始字幕边界，不会把所有字幕粗暴合并。

但核心缺点也很明显：

- 它只能识别“单行内完整出现”的广告表达。
- 只要广告被 ASR 拆成多行，识别能力就明显下降。
- 这类漏判不只是检索纯度问题，还会把广告文本放进知识库候选集合，增加生成阶段“把广告当知识说出来”的风险。

## 3. 思考过程与方案选择

### 3.1 为什么不直接把所有字幕先合并再清洗

可以这样做，但副作用较大：

1. 会破坏原始字幕边界。
2. 清洗完成后再恢复边界比较困难。
3. 切片前的文本结构会变得不可控，不利于后续 chunk 质量稳定。

因此不采用“全文合并后再清洗”的方案。

### 3.2 为什么不直接上模型识别广告

这个场景更适合先用规则法解决：

1. 广告模式相对固定，可解释。
2. 规则法成本低，易调试。
3. 清洗发生在导入链路，应该尽量轻量。
4. 目前问题是跨行漏判，不是复杂语义理解不足。

因此优先采用规则增强，而不是引入额外模型推理。

### 3.3 为什么使用“多行上下文窗口”

本次最终方案的关键点是：

- 输出仍然按行保留字幕边界。
- 判定时不再只看单行，而是允许查看当前行附近的少量上下文。

这里使用的“窗口”不是算法题式的复杂技巧，本质上只是一个工程上的局部上下文扫描：

- 先看当前行。
- 必要时看当前行 + 下一行。
- 再必要时扩展到第三行。

这样做的原因是：

- 广告通常只会被切成 2 到 3 行。
- 局部窗口足以覆盖绝大多数场景。
- 相比全文合并，副作用更小。
- 相比复杂模型，维护成本更低。

## 4. 最终落地方案

最终方案是：

1. 清洗仍然发生在切片前。
2. 输出仍然保留原始字幕行边界。
3. 删除规则升级为“单行优先 + 多行上下文窗口补充”。
4. 所有关键规则全部配置化，并提供默认值。

### 4.1 核心处理流程

最终的处理顺序如下：

1. 从 `Document` 中提取 `Transcript:` 后的字幕正文。
2. 按换行切成字幕段。
3. 逐行标准化。
4. 遍历每一行，先做单行判定。
5. 如果当前起始行本身可疑，再向后做 2 行窗口判定。
6. 如果仍未足够确认，且配置允许，则继续扩展到 3 行或更大窗口。
7. 命中广告后，删除该窗口内的原始字幕行。
8. 对剩余字幕做相邻去重。
9. 将清洗后的文本继续交给 `TokenTextSplitter` 切片。

### 4.2 为什么“只有当前起始行可疑才扩窗”

这是本次实现里一个非常关键的工程细节。

最初如果简单地把 `当前行 + 下一行` 直接拼起来判定，会产生一个问题：

- 正常行本身没问题。
- 下一行是广告。
- 拼接后整体命中广告规则。
- 结果把正常行也一起误删。

所以最终实现里增加了一层约束：

- 只有当前行本身已经表现出“可疑信号”，才允许继续向后扩窗。

这样做的目的是降低误杀率。

### 4.3 为什么还要处理重叠弱关键词

例如：

- `转转`
- `转转二手`

如果简单做 `contains` 统计，这一行里出现 `转转二手` 时，很可能同时把 `转转` 也算进去，导致弱关键词命中数虚高，从而误删。

为了解决这个问题，最终实现采用了：

- 长词优先排序。
- 命中后对已匹配文本做遮罩。
- 后续短词不再重复命中同一段文本。

这个细节是为了避免“重叠关键词双重计数”的误判。

## 5. 配置设计

本次将字幕清洗的关键项全部收敛到：

- `rag.subtitle-cleaning.max-window-size`
- `rag.subtitle-cleaning.conditional-expand-to-3`
- `rag.subtitle-cleaning.weak-keyword-threshold`
- `rag.subtitle-cleaning.strong-patterns`
- `rag.subtitle-cleaning.weak-keywords`
- `rag.subtitle-cleaning.suspicious-patterns`

对应的本地实现文件是：

- `rag-bilibili-server/src/main/java/com/example/ragbilibili/config/SubtitleCleaningProperties.java`
- `rag-bilibili-server/src/main/java/com/example/ragbilibili/config/SubtitleCleaningConfig.java`

### 5.1 各配置项含义

#### `max-window-size`

表示广告联判时允许查看的最大连续行数。

例如：

- `1` 只看单行
- `2` 最多看两行
- `3` 最多看三行

默认值：`3`

#### `conditional-expand-to-3`

表示是否启用“有条件扩窗”。

含义是：

- 不是一开始就固定看 3 行。
- 而是先看单行和 2 行。
- 只有出现足够可疑的迹象时，才继续看第 3 行。

默认值：`true`

#### `weak-keyword-threshold`

表示弱关键词累计命中多少个时，判定为广告。

默认值：`2`

#### `strong-patterns`

强规则正则列表。

只要命中其中任意一个，就直接删除。

默认值里包含如：

- `本[期条次个]视频由.+赞助`
- `感谢.+对本[期条次个]视频的支持`
- `点击.{0,8}(下方|下边).{0,8}(链接|购物车)`
- `下载.{0,12}(app|APP|软件|应用)`

#### `weak-keywords`

弱关键词列表。

单独命中时未必删除，但会参与累计计数。

默认值里包含如：

- `转转`
- `转转二手`
- `赞助`
- `链接`
- `优惠`
- `邀请码`

#### `suspicious-patterns`

可疑信号规则。

它本身不直接决定删除，而是决定“是否值得继续向后扩窗”。

默认值里包含如：

- `感谢`
- `赞助`
- `点击.{0,8}(下方|下边)?`
- `链接`
- `下载`
- `转转`

## 6. 本地代码落点

### 6.1 配置属性类

本地新增：

- `rag-bilibili-server/src/main/java/com/example/ragbilibili/config/SubtitleCleaningProperties.java`

职责：

- 对字幕清洗配置做统一收口。
- 提供默认值。
- 提供 getter/setter。
- 对 `maxWindowSize` 和 `weakKeywordThreshold` 做最小值保护。

### 6.2 配置装配类

本地新增：

- `rag-bilibili-server/src/main/java/com/example/ragbilibili/config/SubtitleCleaningConfig.java`

职责：

- 使用 `@EnableConfigurationProperties(SubtitleCleaningProperties.class)` 接入 Spring 配置绑定。

### 6.3 清洗器实现

本地实现位于：

- `rag-bilibili-server/src/main/java/com/example/ragbilibili/transformer/SubtitleCleaningTransformer.java`

这里的关键方法包括：

- `cleanDocument`：拆出 header 与 transcript，执行清洗并回写 metadata
- `splitSegments`：按行切字幕并标准化
- `cleanTranscriptSegments`：主遍历逻辑
- `findDropWindowSize`：根据当前起始行决定是否扩窗以及删除多少行
- `shouldDropWindow`：判断窗口是否应删除
- `shouldExpandWindow`：判断当前窗口是否足够可疑，值得扩窗
- `countWeakHits`：计算弱关键词命中数
- `maskKeyword`：避免重叠关键词重复计数

### 6.4 导入链路中的位置

本地调用链仍然保持不变：

- `VideoServiceImpl.importVideo(...)` 先调用字幕清洗器
- 再判断清洗后是否仍有有效字幕
- 最后再交给 `TokenTextSplitter`

也就是说，这次优化没有破坏原有“切片前清洗”的总体架构，只是增强了清洗器本身的识别能力和可配置性。

## 9. 方案价值

从工程角度看，这次优化的价值主要体现在以下几方面：

1. 提升切片前字幕清洗的召回能力，减少跨行广告漏判。
2. 保持原始字幕边界，避免为了清洗而破坏后续切片质量。
3. 通过“可疑起始行才扩窗”的约束控制误删。
4. 通过配置化设计降低后续调参成本。
5. 通过测试覆盖把策略稳定下来，而不是只凭经验改规则。
6. 最重要的是，把广告在进入向量库之前清掉，避免它们成为 RAG 的候选知识片段，从而降低模型把广告话术当成视频知识输出的风险。

## 11. 总结

本次优化不是简单“多加几个正则”，而是把字幕清洗从“单行规则”升级成了“可配置的局部上下文检测策略”。

它解决的是一个真实的工程问题：

- 数据进入向量库之前的质量控制
- 自动字幕跨行切分导致的规则失效
- 广告文本进入向量库后造成的知识污染风险
- 清洗效果、误删风险和维护成本之间的平衡

因此，这项工作可以作为一个比较扎实的工程亮点来讲，重点不在“用了滑动窗口”这几个字，而在于：

- 你识别到了问题的根源
- 你给出了成本合理、可解释、可配置、可验证的解决方案
- 你把方案真正落在了本地代码和测试里
