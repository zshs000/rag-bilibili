# 字幕降噪 V2 方案：规则快筛 + LLM 精判

## 1. 背景与问题

B 站视频的自动字幕（ASR）中经常夹杂广告口播、求三连话术等噪声内容。这些内容如果直接进入向量库，会污染 RAG 检索结果，降低问答质量。

### V1 现状

当前采用纯启发式方案（`SubtitleCleaningTransformer`），核心机制：

- **强模式正则**：10 条硬编码正则，命中即删
- **弱关键词阈值**：22 个弱关键词，单行命中 ≥ 2 个才删
- **滑动窗口扩判**：当前行可疑时，拼接 2~3 行联合匹配

### V1 的不足

| 问题 | 说明 |
|------|------|
| 召回率低 | UP 主换个说法就漏（"这期恰饭的金主是…"、"戳一下简介里的链接"） |
| 泛化差 | 正则和关键词全是中文硬编码，无法处理新型话术 |
| 窗口机制复杂但效果有限 | 用"拼字符串"模拟"理解语义"，本质上在赌措辞刚好被正则兜住 |
| 误删风险 | "感谢"、"关注"等弱关键词在正常内容里也很常见 |

### V2 目标

采用**两级过滤架构**：规则层做确定性快筛（零成本），LLM 层做语义级精判（补上规则层的语义理解短板），兼顾清洗精度与调用成本。

## 2. 整体架构

```
字幕原文（按行）
    │
    ▼
┌──────────────────────────┐
│  第一级：规则快筛          │  零 API 成本
│  - 强模式正则 → 直接标记删除 │
│  - 明确的弱关键词组合 → 删除 │
│  输出：kept / dropped      │
└──────────┬───────────────┘
           │
           │ kept 段落
           ▼
┌──────────────────────────┐
│  第二级：LLM 语义精判       │  按需调用 API
│  - 对规则层保留的段落做批量判定│
│  - 按段落分组，一次 prompt   │
│  输出：content / noise      │
└──────────┬───────────────┘
           │
           │ content 段落
           ▼
      清洗后的字幕文档
```

### 为什么不全交给 LLM？

1. **成本**：一个视频可能有几百行字幕，全丢给 LLM 太浪费
2. **延迟**：视频导入是同步流程，多一次 LLM 调用就多几秒
3. **确定性**：像"本期视频由XX赞助"这种模板化广告，规则判断 100% 准确，没必要用 LLM

## 3. 第一级：规则层简化

### 3.1 改动要点

**砍掉**：滑动窗口、可疑信号扩判、`suspiciousPatterns` 配置  
**保留**：强模式正则单行匹配、弱关键词阈值单行匹配、相邻重复行去重

理由：窗口扩判本质上是在用字符串拼接模拟语义理解，现在有 LLM 做后盾，规则层不需要做这个"越级"的事。

### 3.2 简化后的配置

```yaml
rag:
  subtitle-cleaning:
    # 规则层配置（第一级）
    weak-keyword-threshold: 2
    strong-patterns:
      - "本[期条次个]视频由.+赞助"
      - "感谢.+对本[期条次个]视频的支持"
      - "点击.{0,8}(下方|下边).{0,8}(链接|购物车)"
      - "下载.{0,12}(app|APP|软件|应用)"
      - "记得.{0,8}(一键三连|三连|点赞|投币|收藏)"
      - "商务合作"
      - "邀请码"
      - "首单立减"
      - "限时优惠"
      - "高价回收"
    weak-keywords:
      - "转转"
      - "转转二手"
      - "赞助"
      - "三连"
      # ... 同现有列表

    # LLM 层配置（第二级）
    llm:
      enabled: true
      batch-size: 50          # 每次发给 LLM 的最大行数
      model: "deepseek-v3.2"  # 复用项目已有模型
      temperature: 0.0        # 分类任务用 0 温度
      max-retries: 2          # LLM 调用失败重试次数
```

### 3.3 删除的配置项

```yaml
# 以下配置将移除
max-window-size: 3            # 不再需要
conditional-expand-to-3: true # 不再需要
suspicious-patterns: [...]    # 不再需要
```

## 4. 第二级：LLM 语义精判

### 4.1 设计思路

规则层过滤后，剩余段落**分批**发给 LLM，让 LLM 判断每一行是"正文内容"还是"广告/噪声"。

关键设计决策：

| 决策 | 选择 | 原因 |
|------|------|------|
| 输入格式 | 带行号的逐行文本 | LLM 可以精确标记哪几行是广告 |
| 输出格式 | JSON 数组，返回应删除的行号 | 结构化输出，解析可靠 |
| 分批策略 | 每批最多 50 行 | 控制单次 prompt 长度，避免上下文截断 |
| 温度 | 0.0 | 分类任务需要确定性输出 |
| 失败回退 | LLM 调用失败时跳过精判，只用规则层结果 | 保证可用性，不因 LLM 故障阻塞导入 |

### 4.2 Prompt 设计

```text
你是一个视频字幕内容分类器。以下是一段 B 站视频的自动字幕，每行前面有行号。

请判断哪些行属于以下噪声类型，返回应该删除的行号列表：
1. 广告口播（赞助商推广、产品推荐、优惠活动）
2. 求互动话术（求点赞、投币、收藏、转发、关注）
3. 无意义填充（ASR 识别错误导致的乱码、纯语气词重复）

注意：
- 只标记明确的噪声行，不确定的保留
- "感谢观看"、"下期再见"等结束语属于正常内容，不要删除
- 视频内容中讨论广告、营销等话题时，这些是正文，不要删除

字幕内容：
---
{numbered_lines}
---

请以 JSON 格式返回应删除的行号数组，例如：{"drop": [3, 4, 5]}
如果没有需要删除的行，返回：{"drop": []}
```

### 4.3 核心类设计

```
com.example.ragbilibili.transformer/
├── SubtitleCleaningTransformer.java     ← 改造：第一级规则层（简化）
├── LlmSubtitleFilter.java              ← 新增：第二级 LLM 精判
└── SubtitleCleaningPipeline.java        ← 新增：编排两级流程
```

#### SubtitleCleaningPipeline（编排层）

```java
@Component
public class SubtitleCleaningPipeline implements DocumentTransformer {

    private final SubtitleCleaningTransformer ruleCleaner;
    private final LlmSubtitleFilter llmFilter;
    private final SubtitleCleaningProperties properties;

    @Override
    public List<Document> apply(List<Document> documents) {
        // 第一级：规则快筛
        List<Document> ruleFiltered = ruleCleaner.apply(documents);

        // 第二级：LLM 精判（可配置开关）
        if (properties.getLlm().isEnabled()) {
            return llmFilter.apply(ruleFiltered);
        }

        return ruleFiltered;
    }
}
```

#### LlmSubtitleFilter（LLM 判定层）

```java
@Component
public class LlmSubtitleFilter implements DocumentTransformer {

    private final ChatClient.Builder chatClientBuilder;
    private final SubtitleCleaningProperties properties;

    @Override
    public List<Document> apply(List<Document> documents) {
        return documents.stream()
                .map(this::filterDocument)
                .collect(Collectors.toList());
    }

    private Document filterDocument(Document document) {
        // 1. 提取 Transcript 部分的字幕行
        List<String> segments = extractSegments(document);
        if (segments.isEmpty()) return document;

        // 2. 分批调用 LLM
        Set<Integer> dropLines = new HashSet<>();
        List<List<String>> batches = partition(segments, properties.getLlm().getBatchSize());

        int lineOffset = 0;
        for (List<String> batch : batches) {
            Set<Integer> batchDrops = callLlmForBatch(batch, lineOffset);
            dropLines.addAll(batchDrops);
            lineOffset += batch.size();
        }

        // 3. 过滤掉 LLM 标记的噪声行，重组文档
        List<String> kept = new ArrayList<>();
        for (int i = 0; i < segments.size(); i++) {
            if (!dropLines.contains(i)) {
                kept.add(segments.get(i));
            }
        }

        return rebuildDocument(document, kept);
    }

    private Set<Integer> callLlmForBatch(List<String> batch, int lineOffset) {
        String numberedLines = buildNumberedLines(batch, lineOffset);
        String prompt = buildPrompt(numberedLines);

        try {
            ChatClient chatClient = chatClientBuilder.build();
            String response = chatClient.prompt()
                    .user(prompt)
                    .call()
                    .content();
            return parseDropLines(response);
        } catch (Exception e) {
            log.warn("LLM 字幕精判调用失败，跳过本批次: {}", e.getMessage());
            return Set.of();  // 失败时不删任何行，保守策略
        }
    }
}
```

## 6. 调用流程（时序）

```
VideoServiceImpl.importVideo()
    │
    ├─ BilibiliDocumentReader.get()        → 获取原始字幕文档
    │
    ├─ SubtitleCleaningPipeline.apply()    → 两级清洗
    │   │
    │   ├─ SubtitleCleaningTransformer     → 第一级：规则快筛
    │   │   ├─ 强模式正则单行匹配 → dropped
    │   │   ├─ 弱关键词阈值单行匹配 → dropped
    │   │   └─ 相邻重复行去重
    │   │
    │   └─ LlmSubtitleFilter              → 第二级：LLM 精判
    │       ├─ 提取规则层保留的字幕行
    │       ├─ 按 batch-size 分批
    │       ├─ 构造 prompt，调 ChatClient
    │       ├─ 解析 JSON 响应，得到 drop 行号
    │       └─ 过滤噪声行，重组文档
    │
    ├─ hasUsableSubtitleContent()          → 检查是否还有有效内容
    ├─ TokenTextSplitter.apply()           → 切分
    └─ DashVectorStore.add()              → 入库
```

## 7. 成本估算

以一个典型 10 分钟 B 站视频为例：

| 阶段 | 字幕行数 | 说明 |
|------|---------|------|
| 原始字幕 | ~200 行 | ASR 按句切分 |
| 规则层过滤后 | ~180 行 | 约 10% 被强模式命中 |
| 发给 LLM | ~180 行 | 4 批 × 50 行（实际可能 3~4 批） |

每批约 500~800 tokens（输入） + 50 tokens（输出），总计约 **3000~4000 tokens/视频**。

以 DeepSeek V3 的定价（输入 ¥1/M tokens，输出 ¥2/M tokens），单个视频清洗成本约 **¥0.003~0.005**，几乎可以忽略。

## 8. 容错与降级

| 场景 | 处理策略 |
|------|----------|
| LLM 调用超时 | 重试 `maxRetries` 次后跳过，只用规则层结果 |
| LLM 返回非法 JSON | 跳过本批次，记录 warn 日志 |
| LLM 服务不可用 | `llm.enabled=false` 一键降级为纯规则模式 |
| LLM 误删正文 | prompt 中强调"不确定的保留"，宁可漏删不误删 |
