# RAG-Bilibili 向量化实现说明

## 一、文档目的

本文档用于说明 `RAG-Bilibili` 项目中 B 站视频内容从 `reader` 输出到向量库写入的实现思路，重点澄清以下问题：

- Bilibili `reader` 会返回什么数据结构
- 哪一个 `Document` 才是实际参与 RAG 的内容文档
- 文档切片后会保留哪些 metadata
- 业务侧需要补充哪些 metadata
- 自定义向量 `id` 如何进入 DashVector
- 为什么本方案无需修改 `reader` 和 `DashVectorStore` 的内部实现

本文档属于实现说明/技术设计说明，不替代系统需求规约。

## 二、总体结论

当前方案可以在**不修改**以下两个组件内部实现的前提下完成目标：

- `spring-ai-alibaba-starter-document-reader-bilibili`
- `spring-ai-alibaba-starter-dashvector-store`

原因如下：

- `reader` 已经能返回带 metadata 的 `Document`
- `DashVectorStore` 已经支持读取 `Document.id` 作为 Dash 文档主键
- `DashVectorStore` 已经支持将 `Document.metadata` 写入 Dash 的 `fields`
- `DashVectorStore` 已经支持将 Spring AI 的过滤表达式转换为 Dash 查询过滤语句

因此，当前项目只需要在**业务层**补齐“过滤文档、切片、补 metadata、补 id、入库”这条链路即可。

## 三、Bilibili Reader 输出说明

### 1. Reader 返回值

Bilibili `reader` 的 `get()` 返回值是 `List<Document>`。

虽然返回类型是列表，但对单个视频而言，当前实现会返回两个核心 `Document`：

- 一个是视频信息文档
- 一个是视频内容文档

### 2. 视频信息文档

该文档用于表达视频基础信息，metadata 包括：

- `bvid`
- `document_type=metadata`
- `title`
- `description`

该文档主要适合作为辅助信息文档，不建议直接作为 RAG 主检索内容写入向量库。

### 3. 视频内容文档

该文档是后续 RAG 的主要来源。metadata 包括：

- `bvid`
- `document_type=content`
- `title`

其 `text` 内容为组合文本，包含：

- 视频标题
- 视频简介
- 字幕正文

因此，在进入切片流程之前，业务侧应从 `reader` 返回的文档列表中，筛选出 `document_type=content` 的文档作为真正参与知识切分与向量化的输入。

## 四、业务层处理原则

### 1. 仅处理内容文档

业务层应只保留 `document_type=content` 的 `Document`。

虽然过滤后的结果仍然是 `List<Document>`，但在单视频场景下，实际只会剩下一个内容文档。

### 2. 删除无关 metadata

在进入切片前，可删除当前业务不需要继续携带的 metadata，例如：

- `document_type`

保留的 metadata 应至少包括：

- `bvid`
- `title`

其中：

- `bvid` 用于单视频过滤与视频归属标识
- `title` 用于结果展示、引用展示与排查

## 五、切片后的 Document 说明

### 1. 切片组件与结果类型

本方案采用 Spring AI 提供的 `TokenTextSplitter` 进行切片，按固定 token 窗口切分文本，并按业务需要配置分片大小、最小分片字符数、最小可嵌入长度、最大分片数量以及是否保留分隔符等参数。

内容文档经过 `TokenTextSplitter` 切片后，结果仍然是 `List<Document>`。

也就是说，切片不会把文档变成其他类型对象，而是会生成多个更小的 `Document`。

### 2. 切片后保留的数据

由于 `TokenTextSplitter` 属于 Spring AI 的文档切片组件，切片后的每个小 `Document` 仍然携带原始内容文档的核心 metadata。

在本方案中，至少保留以下字段：

- `bvid`
- `title`

因此，每一个切片都天然能够知道自己属于哪个视频、对应哪个标题。

**说明**：chunk_index 表示分片序号，不代表句子。切片是按固定 token 窗口切分的，不是按句子切分。

### 3. 业务层新增 metadata

切片完成后，业务层还需要为每一个小 `Document` 手动补充以下 metadata：

- `user_id`
- `chunk_index`
- `total_chunks`

**类型转换说明**：
- 数据库中这些字段的类型：user_id (BIGINT)、chunk_index (INT)、total_chunks (INT)
- 写入向量库时，统一转换为 String 类型
- 转换规则：直接调用 `String.valueOf()` 或 `toString()` 方法
- 原因：DashVector 的 fields 统一按字符串处理，避免类型不一致导致的过滤问题

这些字段的作用如下：

- `user_id`
  - 用于多用户隔离
  - 用于全库问答时仅检索当前用户的数据
- `chunk_index`
  - 用于表示当前分片序号
  - 用于构造自定义向量 `id`
  - 用于问题排查与分片定位
- `total_chunks`
  - 用于表示同一视频总分片数
  - 用于前端展示、排查和后续扩展

**命名规范说明**：
- Java 代码中使用驼峰命名：userId, chunkIndex, totalChunks
- 向量 metadata 使用下划线命名：user_id, chunk_index, total_chunks
- 数据库列使用下划线命名：user_id, chunk_index, total_chunks

## 六、最终入向量库的 metadata 约定

每一个切片后的小 `Document`，最终应至少携带以下 metadata：

- `user_id`
- `bvid`
- `title`
- `chunk_index`
- `total_chunks`

其中：

- `bvid`、`title` 来源于 `reader`
- `user_id`、`chunk_index`、`total_chunks` 来源于业务层补充

**类型说明**：
- 所有字段在写入向量库时统一转换为 String 类型
- 数据库中的数值类型（BIGINT、INT）需要通过 `String.valueOf()` 转换
- 这样可以降低字段类型不一致带来的过滤问题

## 七、自定义向量 ID 设计

### 1. ID 规则

每一个切片后的 `Document`，都需要在业务层显式设置自定义 `id`。

推荐规则如下：

- `user_id_bvid_chunk_index`

示例：

- `1001_BV1DCfsBKExV_2`

### 2. 为什么必须设置 Document.id

当前项目删除向量时，依赖 DashVector 的按 `id` 删除能力，因此必须确保每个向量的 `id` 是业务可控、可回溯、可批量定位的。

如果不自行设置 `Document.id`，则无法满足以下要求：

- 视频删除时精确删除对应向量
- 本地 MySQL 建立稳定的分片与向量映射
- 排查某个分片对应的向量记录

## 八、DashVectorStore 的配合机制

### 1. Document.id 的作用

`DashVectorStore` 在写入时，会读取每个 `Document` 的 `id`。

因此，只要业务层在切片后构造好每个小 `Document` 的 `id`，则该 `id` 会直接成为 Dash 中真正的文档主键。

### 2. metadata 的作用

`DashVectorStore` 在写入时，会将 `Document.metadata` 逐项写入 Dash 的 `fields`。

因此，以下 metadata 最终都可以进入 Dash：

- `user_id`
- `bvid`
- `title`
- `chunk_index`
- `total_chunks`

这使得后续检索时可以基于这些字段执行过滤。

### 3. 分片正文的作用

`DashVectorStore` 在写入时，还会把切片文本写入 `content` 字段。

因此，Dash 中最终同时拥有：

- 向量本身
- 文档主键 `id`
- 业务 metadata fields
- 分片正文 `content`

其中，业务 metadata fields 与分片正文 `content` 最终都以 Dash 的 `fields` 形式写入。

这意味着系统不仅能做相似检索，还能在命中后直接取回片段正文，用于：

- RAG 上下文拼接
- 前端展示命中片段
- 来源定位

## 九、检索过滤设计

### 1. 单视频对话

单视频对话时，应至少基于以下字段做联合过滤：

- `user_id`
- `bvid`

这样可以保证：

- 只检索当前用户自己的数据
- 只检索当前用户选中的那一个视频

### 2. 全部视频对话

全库对话时，应至少基于以下字段过滤：

- `user_id`

这样可以保证：

- 在共享向量库中，不会检索到其他用户的数据

### 3. 为什么无需改 DashVectorStore

因为当前 `DashVectorStore` 已具备以下能力：

- 将 `metadata` 写入 `fields`
- 将 Spring AI `filterExpression` 转换为 Dash 的过滤语句

所以当前项目只需在业务层保证写入的 metadata 完整、命名统一、类型统一即可。

### 3.1 DashVector 适配说明

在这一层适配中，我已经将 DashVector 与 Spring AI 的通用向量检索语义进行了对齐，因此业务层可以直接使用 Spring AI 抽象，而不需要再自行处理 Dash 的底层差异。

我已经完成的适配主要包括：

- 将 `Document.metadata` 写入 Dash 的 `fields`
- 将 Spring AI 的 `filterExpression` 转换为 Dash 可执行的过滤语句
- 将查询文本先转换为查询向量，再发起 Dash 相似检索
- 将 Dash 返回的分数语义转换为 Spring AI 可直接使用的相似度语义
- 在内部应用 `similarityThreshold`，业务层无需再次自行转换或重复判断

因此，在业务侧使用时，只需要按照 Spring AI 的标准方式传入：

- 查询文本
- `topK`
- `filterExpression`
- `similarityThreshold`

即可直接复用该适配层完成 DashVector 的相似检索。

### 4. 过滤转换说明

本方案的过滤能力分为上下两层：

- 上层使用 Spring AI 统一的 `filterExpression`
- 底层由 DashVector 执行基于 `fields` 的过滤

二者之间的衔接由 `DashVectorFilterExpressionConverter` 完成。

具体转换逻辑如下：

- 写入阶段
  - `Document.metadata` 中的字段会逐项写入 Dash 的 `fields`
  - 例如 `user_id`、`bvid`、`title`、`chunk_index`、`total_chunks` 都会成为 Dash 中可过滤的字段
- 查询阶段
  - 上层传入 Spring AI 的 `filterExpression`
  - `DashVectorStore` 会先调用转换器，将其转换为 Dash 可接受的过滤字符串
  - 然后将该过滤字符串放入 Dash 查询请求中执行

字段名的转换规则非常直接：

- Spring AI 过滤表达式中的字段名，直接使用 metadata 中的 key 名称
- 不做额外的字段路径包装
- 不做字段名重命名

例如：

- 若 metadata 中写入的是 `user_id`
- 且查询时使用的过滤字段也是 `user_id`
- 则最终会直接按 Dash 的对应 `fields` 字段进行过滤

该转换器当前支持的主要操作包括：

- 等于
- 不等于
- 大于
- 大于等于
- 小于
- 小于等于
- `AND`
- `OR`
- `IN`
- `NIN`

其中：

- 字符串值会自动加引号并转义
- `IN` 和 `NIN` 会转换为多个等值判断的组合表达式

因此，在本项目中可以直接采用 Spring AI 的统一过滤表达方式，而不需要在业务层自行拼接 Dash 原生过滤语句。

前提只有一个：

- 写入 metadata 时使用的字段名，与查询过滤时使用的字段名必须完全一致

对于本项目，建议统一使用：

- `user_id`
- `bvid`
- `title`
- `chunk_index`
- `total_chunks`

## 十、MySQL 映射关系说明

虽然向量的主键 `id` 已经可以通过业务规则直接构造，但本项目仍然需要在 MySQL 中维护映射关系。

至少应记录：

- 用户 ID
- 视频标识
- 分片序号
- 分片文本
- 向量 ID

原因如下：

- 删除视频时需要先查出全部向量 `id`
- MySQL 是删除链路的权威数据来源
- 便于后续做批量删除、补偿处理和问题排查

## 十一、最终流程总结

完整实现流程如下：

1. 通过 Bilibili `reader` 读取视频，得到两个 `Document`
2. 从中筛选出 `document_type=content` 的内容文档
3. 删除当前业务不再需要的 metadata，仅保留 `bvid`、`title`
4. 使用基于 Spring AI 的 `TokenTextSplitter` 对内容文档进行切片
5. 为每个切片后的小 `Document` 补充：
   - `user_id`
   - `chunk_index`
   - `total_chunks`
6. 为每个切片后的小 `Document` 设置自定义 `id`（格式：user_id_bvid_chunk_index）
7. 将这些 `Document` 写入 DashVector
8. 将视频、分片、向量 `id` 的映射关系写入 MySQL

## 十二、结论

本方案的关键在于：

- 不修改 `reader`
- 不修改 `DashVectorStore`
- 通过业务层对 `Document` 做二次整理和增强

只要业务层完成以下工作：

- 过滤正确的内容文档
- 切片
- 补齐 metadata
- 设置自定义 `id`

即可直接复用现有组件完成向量化、检索、过滤与删除链路。
