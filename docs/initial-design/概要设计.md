# RAG-Bilibili 概要设计

## 一、项目简介

### 1. 目的

本文档是 RAG-Bilibili 项目的概要设计文档，用于指导项目的技术实现。本文档基于《RAG-Bilibili 系统需求规约》，详细说明系统的技术架构、接口设计、数据结构设计和关键业务流程。

### 2. 范围

本文档适用于 RAG-Bilibili 项目第一阶段（V1）的开发工作，涵盖：

- 前后端技术架构设计
- RESTful API 接口定义
- 数据库表结构设计
- 关键业务流程设计
- 前端页面结构设计

### 3. 参考资料

- 《RAG-Bilibili 系统需求规约》
- 《RAG-Bilibili 向量化实现说明》
- Spring Boot 官方文档：<https://spring.io/projects/spring-boot>
- Spring AI 官方文档：<https://docs.spring.io/spring-ai/reference/>
- Spring AI Alibaba 官网：<https://java2ai.com/>
- Vue 3 官方文档：<https://vuejs.org/>
- Element Plus 官方文档：<https://element-plus.org/>

### 4. 概述

RAG-Bilibili 是一个基于 B 站视频字幕内容的检索增强问答系统。系统采用前后端分离架构，前端使用 Vue 3 + Element Plus 构建用户界面，后端使用 Spring Boot 3.x + Spring AI 构建业务逻辑和 AI 能力，使用 MySQL 存储业务数据，使用 DashVector 存储向量数据。

V1 版本默认采用**本机同源部署**方式：前端通过后端同源地址访问，或由后端直接托管前端静态资源；本阶段暂不单独设计跨域访问、跨站认证与分布式部署方案。

系统核心能力包括：
- 用户注册登录与数据隔离
- B 站视频字幕导入与向量化
- 基于 RAG 的单视频/全视频问答
- 多轮对话支持
- 视频管理与删除联动

#### 4.3 发送消息接口（流式）

**请求**：
```json
POST /api/sessions/{id}/messages/stream
{
  "content": "这个视频讲了什么？"
}
```

**响应**：
- Content-Type: `text/event-stream`
- 使用 SSE (Server-Sent Events) 协议流式返回

**流式数据格式**：
```
data: {"type": "start", "userMessageId": 1}

data: {"type": "content", "delta": "这个"}

data: {"type": "content", "delta": "视频"}

data: {"type": "content", "delta": "主要"}

data: {"type": "content", "delta": "讲解了"}

data: {"type": "content", "delta": "..."}

data: {"type": "end", "userMessageId": 1, "assistantMessageId": 2, "fullContent": "这个视频主要讲解了..."}
```

**事件类型说明**：
- `start`：开始生成回答，返回用户消息 ID（已保存到数据库）
- `content`：增量内容片段
- `end`：生成完成，返回用户消息 ID、助手消息 ID（已保存到数据库）和完整内容

**字段说明**：
- `userMessageId`：用户消息的数据库主键 ID，用于前端关联请求和响应
- `assistantMessageId`：助手消息的数据库主键 ID，用于前端展示和后续引用
- `fullContent`：助手回答的完整内容（用于前端校验或重新渲染）

**前端接收示例**：
```javascript
const response = await fetch('/api/sessions/1/messages/stream', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ content: '这个视频讲了什么？' })
});

const reader = response.body.getReader();
const decoder = new TextDecoder();
let shouldStop = false;

while (true) {
  const { done, value } = await reader.read();
  if (done) break;

  const chunk = decoder.decode(value);
  const lines = chunk.split('\n');

  for (const line of lines) {
    if (line.startsWith('data: ')) {
      try {
        const data = JSON.parse(line.slice(6));
        if (data.type === 'content') {
          // 追加显示增量内容
          appendContent(data.delta);
        } else if (data.type === 'end') {
          // 流式完成，设置标志位跳出外层循环
          shouldStop = true;
          break;
        }
      } catch (e) {
        console.error('JSON parse error:', e);
      }
    }
  }

  if (shouldStop) break;
}
```

**数据库保存时机**：
- 用户消息：请求开始时立即保存到数据库，获得 `userMessageId`，在 `start` 事件中返回
- 助手消息：流式生成完成后保存完整内容到数据库，获得 `assistantMessageId`，在 `end` 事件中返回

**前端关联逻辑**：
- 前端发送请求后，从 `start` 事件获取 `userMessageId`，可用于在界面上标识用户消息
- 前端从 `content` 事件逐步拼接助手回答内容
- 前端从 `end` 事件获取 `assistantMessageId` 和 `fullContent`，用于最终确认和存储
- 通过 `userMessageId` 可以关联一次完整的问答交互（一个用户消息 + 一个助手消息）

5. 系统显示导入进度

1. 用户在会话列表页面点击"新建对话"
2. 选择对话类型（单视频/全部视频）

4. 进入对话页面
5. 输入问题并发送
6. 系统显示回答
7. 支持多轮对话

#### 2.3 视频删除流程
1. 用户在视频列表页面点击删除按钮
2. 系统弹出确认对话框
3. 用户确认删除
4. 系统执行删除操作（删除向量、数据库记录、关联会话）
5. 刷新视频列表

## 六、流程设计

### 1. 视频导入流程

```
用户提交导入请求
    ↓
校验用户登录状态
    ↓
解析 BV 号（如果输入是 URL）
    ↓
校验 BV 号是否已存在
    ↓
创建 video 记录，状态设置为 IMPORTING
    ↓
调用 Bilibili Reader 获取视频数据
    ↓
筛选 document_type=content 的文档
    ↓
使用 TokenTextSplitter 切分文本
    ↓
为每个分片补充 metadata (user_id, chunk_index, total_chunks)
    ↓
为每个分片设置自定义 id (user_id_bvid_chunk_index)
    ↓
调用百炼向量模型生成 Embedding
    ↓
写入 DashVector
    ↓
写入 MySQL (chunk, vector_mapping)
    ↓
更新 video 状态为 SUCCESS
    ↓
统计 chunk 数量
    ↓
返回导入成功
```

**状态说明**：
- **IMPORTING**：视频导入中，用户可在列表中看到但不能使用
- **SUCCESS**：导入成功，用户可正常使用
- **FAILED**：导入失败，用户可查看失败原因并删除后重新导入

**失败处理**：
- 任何步骤失败时，更新 video 状态为 FAILED
- 如果向量已写入但数据库写入失败，需清理已写入的向量（V1 已知风险）

**chunkCount 来源**：
- 导入成功后，通过查询 chunk 表统计：`SELECT COUNT(*) FROM chunk WHERE video_id = ?`
- 返回给前端时包含在响应中

### 2. 问答流程（流式响应）

```
用户发送问题
    ↓
校验会话归属
    ↓
保存用户消息到数据库，获得 userMessageId
    ↓
发送 start 事件（包含 userMessageId）
    ↓
根据会话类型构建过滤条件
  - 单视频：user_id + bvid
  - 全视频：user_id
    ↓
从消息表读取历史消息
    ↓
使用 QuestionAnswerAdvisor 执行 RAG 检索
    ↓
调用 OpenAI 兼容接口流式生成回答
    ↓
通过 SSE 逐步返回 content 事件（增量内容片段）
    ↓
流式生成完成
    ↓
保存助手消息到数据库，获得 assistantMessageId
    ↓
发送 end 事件（包含 userMessageId、assistantMessageId、fullContent）
    ↓
关闭 SSE 连接
```

**流程说明**：
- 用户消息在请求开始时立即保存，确保不丢失
- 使用 Spring AI 的 `ChatClient.stream()` 方法实现流式生成
- 通过 SSE 协议将生成的内容片段实时推送给前端
- 流式完成后保存完整的助手消息到数据库

### 3. 视频删除流程

```
用户请求删除视频
    ↓
校验视频归属（确保 video.user_id = userId）
    ↓
从 vector_mapping 表查询所有向量 ID
    ↓
调用 DashVector 批量删除向量（失败则抛异常，整体回滚）
    ↓
删除 session 表中 video_id = videoId 的单视频会话
    ↓
删除 message 表中对应会话的消息
    ↓
删除 vector_mapping 记录
    ↓
删除 chunk 记录
    ↓
删除 video 记录
    ↓
返回删除成功
```

**删除规则说明**：
- **单视频会话**：会话及其所有消息都会被删除
- **全视频会话**：会话保留，历史消息保留
  - 后续检索时不会再命中该视频的内容（因为向量已删除）
  - V1 阶段不在 message 表中记录来源 video_id，也不对历史消息做“来源视频已删除”标记
  - 历史消息仅作为已生成内容保留，不再参与删除视频后的检索命中判断

**失败处理**：
- 如果向量删除失败，整个删除操作失败，数据库记录不删除
- 如果向量删除成功但数据库删除失败，存在数据不一致风险（V1 已知问题）

### 1. 实体关系图 (ER 图)

### 2. 关键关系说明

- **User - Video**：一对多，一个用户可以导入多个视频
- **Video - Chunk**：一对多，一个视频可以切分为多个分片
- **Chunk - VectorMapping**：一对一，一个分片对应一个向量映射
- **User - Session**：一对多，一个用户可以创建多个会话
- **Video - Session**：一对多（可选），单视频会话关联一个视频
- **Session - Message**：一对多，一个会话包含多条消息

## 八、范式分析

### 1. 范式符合性

本系统数据库设计符合**第三范式（3NF）**：

- **第一范式（1NF）**：所有表的字段都是原子性的，不可再分
- **第二范式（2NF）**：所有非主键字段完全依赖于主键
- **第三范式（3NF）**：所有非主键字段不存在传递依赖

### 2. 设计原则

- **禁止外键约束**：遵循阿里巴巴开发规范，严禁使用数据库外键约束，数据一致性应在业务层保证。外键约束会影响数据库性能，增加维护复杂度，且在分库分表场景下难以维护
- **冗余存储优化**：分片表冗余存储 `bvid`、`title`、`total_chunks` 字段，避免频繁 JOIN 查询，提升性能
- **数据隔离性**：所有业务表都包含 `user_id`，确保用户数据隔离
- **可追溯性**：关键操作记录 `create_time`，便于审计和排查
- **可扩展性**：预留 `status` 等字段，便于后续功能扩展
- **业务层约束**：通过 Service 层逻辑保证数据一致性，包括：
  - 删除视频时级联删除分片、向量映射、会话等关联数据
  - 创建分片时确保 video_id 存在
  - 创建会话时确保 video_id（如果有）存在

### 3. 反范式化设计

本系统已采用适度反范式化设计以提升查询性能：

**已实现的冗余存储**：
- **Chunk 表**：冗余存储 `bvid`、`title`、`total_chunks`
  - 避免查询分片时频繁 JOIN video 表
  - 与向量库 metadata 保持一致，便于数据同步
  - 在重建向量索引时无需额外查询

**未来可考虑的优化**：
- **Session 表**：可冗余 `video_title`，避免会话列表查询时 JOIN video 表
- **Message 表**：可冗余 `session_type`，便于按会话类型统计和查询

**数据一致性保证**：
- 冗余字段的更新由 Service 层统一管理
- 视频标题修改时（如果支持），需同步更新 chunk 表中的 title 字段
- 通过事务保证冗余数据的一致性
