# RAG-Bilibili 系统需求规约

## 一、项目简介

### 1. 目的

RAG-Bilibili 项目旨在面向普通用户提供基于 B 站视频字幕内容的检索增强问答能力，解决以下典型问题：

- 用户没有时间完整观看或阅读视频内容，希望快速获取视频核心信息。
- 用户已经看过部分视频，但希望后续能够围绕视频内容进行回顾、提问和记忆强化。
- 用户希望对自己已导入的视频知识建立可持续复用的个人知识库，并基于该知识库进行多轮问答。

本系统通过导入 B 站视频字幕信息，构建固定粒度向量索引，并结合大模型完成面向单视频或多视频范围的问答服务。

### 2. 范围

本规约适用于 `RAG-Bilibili` 项目第一阶段（V1）建设，范围包括：

- 用户注册、登录及用户数据隔离。
- 基于 B 站账号凭证与视频标识的字幕信息导入。
- 基于字幕内容的固定粒度切分、向量化、索引存储与映射管理。
- 基于单视频或全部已导入视频的数据范围进行检索增强问答。
- 多轮对话能力。
- 已导入视频列表查看与删除。
- 删除视频时联动删除数据库映射、向量数据及相关单视频会话。

本阶段**不包含**以下内容：

- 复杂 RAG 流程编排，如重排、知识图谱、查询改写、多路召回等高级增强链路。
- 额外的无字幕视频处理链路；当前依赖 `reader` 组件完成字幕获取，只要视频中存在可识别语音，原则上即可产出字幕；若最终仍无法形成有效字幕，通常意味着视频本身没有语音内容，或语音内容无法被识别。
- B 站收藏夹、UP 主页、投稿列表等批量导入能力。
- 后台管理平台、审核平台、运营分析平台。
- 按条件直接删除 DashVector 向量数据。

### 4. 概述

本系统为前后端分离应用，前端采用 Vue，后端采用 Spring Boot + Spring AI，并结合 Spring AI Alibaba 相关组件完成 B 站字幕读取、百炼向量生成、DashVector 向量存储以及基于 OpenAI 兼容接口的对话模型调用。系统同时使用 MySQL 存储业务数据，包括用户数据、视频基础信息、字幕分片信息、向量 ID 映射关系、会话数据及消息记录，以支撑用户隔离、数据管理、向量删除联动和会话持久化等核心能力。

系统主要流程如下：

1. 用户注册并登录系统。
2. 用户填写 B 站账号凭证信息及视频标识信息。
3. 系统通过 Bilibili Reader 获取视频字幕及视频基础信息。
4. 系统对字幕进行固定粒度切分（基于 token 窗口）。
5. 系统调用阿里云百炼向量模型生成 Embedding。
6. 系统将分片向量写入 DashVector，并将视频、分片、向量 ID 映射写入 MySQL。
7. 用户创建新对话时选择问答范围：单视频或全部视频。
8. 系统通过 `QuestionAnswerAdvisor` 完成检索增强问答，并支持多轮对话。
9. 用户删除视频时，系统同步删除数据库映射、DashVector 向量以及相关单视频对话。

### 5. 名词解释

- **B 站**：指哔哩哔哩（Bilibili）视频平台。
- **BV号**：B 站视频唯一标识之一。
- **视频 URL**：B 站视频完整访问地址，系统可从中提取 BV 号。
- **字幕**：B 站视频的文本字幕内容，为本系统主要知识来源。
- **RAG**：Retrieval-Augmented Generation，检索增强生成。
- **Embedding**：向量化表示，用于将文本转为可检索的语义向量。
- **DashVector**：阿里云向量数据库服务，用于存储与检索文本向量。
- **QuestionAnswerAdvisor**：Spring AI 中用于实现基础检索增强问答的组件。
- **分片**：将视频字幕按固定 token 窗口切分后形成的最小检索单元。chunk_index 表示分片序号，不代表句子。
- **向量 ID**：写入 DashVector 时使用的业务自定义唯一标识。

## 二、需求分析

### 1. 需求概述

系统面向普通用户提供“导入视频—构建知识—围绕视频对话”的完整闭环能力。用户通过输入个人 B 站账号凭证与视频标识信息导入指定视频的字幕数据，并以个人知识库方式保存。这里的视频标识信息仅支持 `BV号` 或包含 `BV号` 的 Bilibili 视频 URL；如果用户输入的是 URL，则后端需能够从 URL 中解析出 `BV号`，例如：`https://www.bilibili.com/video/BV1DCfsBKExV`。随后用户可选择针对某一个视频或全部已导入视频进行多轮问答。

系统核心目标如下：

- 让用户在不完整观看视频的情况下快速获得视频知识。
- 让用户能够对自己关心的视频内容进行持续提问和记忆强化。
- 让视频知识以个人私有数据方式沉淀，避免不同用户之间的数据混淆。
- 通过业务自定义向量 ID 与 MySQL 映射关系，确保删除操作可精确、可控、可追踪。

### 2. 功能需求分析

#### 2.1 用户与认证模块

##### 2.1.1 用户注册

- 系统应支持用户注册。
- 注册后系统应创建独立用户数据空间。
- 用户之间的数据必须逻辑隔离。

##### 2.1.2 用户登录

- 系统应支持用户登录。
- 用户登录成功后方可进行视频导入、视频问答、视频管理与会话管理。
- 系统应基于登录态识别当前操作所属用户。

##### 2.1.3 用户数据隔离

- 一个用户导入的视频、分片、向量映射、会话记录不得被其他用户访问。
- 所有与视频、分片、会话、向量映射相关的查询和操作均应带有用户维度校验。

#### 2.2 视频导入模块

##### 2.2.1 导入入口

- 系统仅支持以下两种视频导入方式：`BV号` 或包含 `BV号` 的 `Bilibili 视频URL`。
- 若输入为视频 URL，则该 URL 必须能够被后端解析出 `BV号`，例如：`https://www.bilibili.com/video/BV1DCfsBKExV`
- 系统当前不支持通过收藏夹、UP 主页、投稿列表、搜索关键词等方式导入。后续版本可能支持相关扩展能力，但由于涉及版权与平台合规边界，本阶段先仅支持单视频导入。

##### 2.2.2 账号凭证输入

- 用户导入视频时，应输入以下 B 站账号凭证字段：
  - `SESSDATA`
  - `bili_jct`
  - `buvid3`
- 以上三个字段均为必填字段。
- 系统应将以上字段用于调用 Bilibili Reader 获取视频信息和字幕信息。
- 由于账号凭证属于高度敏感隐私数据，一旦泄露可能直接导致账号风险，因此系统不得在后端数据库中持久化保存上述凭证。
- 后端仅可在当前导入请求处理过程中临时使用凭证，不得落库，不得写入缓存，不得输出到日志。
- 凭证应仅保存在前端本地，用于用户后续再次导入时复用。
- 前端默认应采用 `sessionStorage` 保存凭证信息，以保证页面刷新后凭证仍可复用，同时在浏览器标签页关闭后自动失效，降低长期暴露风险。

##### 2.2.3 导入内容范围

- 系统导入的数据范围应至少包括：
  - 视频标题
  - 视频简介
  - 视频字幕正文
  - 视频 BV 号
- 系统应保留可用于检索和展示的基础元数据。
- 其中 `BV号` 必须作为 metadata field 一并写入向量数据库，以支持按单个视频范围进行检索与对话。
- 同时，`用户ID` 也必须作为 metadata field（字段名为 `user_id`）一并写入向量数据库，以支持在共享向量库场景下进行用户级数据隔离。

##### 2.2.4 字幕可用性约束

- 系统以字幕为核心知识来源。
- 系统依赖 `reader` 组件获取字幕；按照当前能力，只要视频中存在可识别语音，原则上都应能够获得字幕内容。
- 仅当视频没有语音内容，或语音内容无法被识别时，系统才可能无法形成有效字幕。
- 当系统无法形成有效字幕时，应提示用户当前视频暂不可用于本系统问答。
- 本阶段不单独处理真正无声音的视频；若视频本身无语音内容，则不属于本系统的有效知识来源。

##### 2.2.5 重复导入规则

- 当用户尝试导入自己已导入过的同一 `BV号` 视频时，系统应提示“该视频已存在，请先删除后再导入”。
- 系统不应对已存在视频执行自动覆盖重建。
- 系统不应在重复导入时自动删除旧向量或重建索引。

#### 2.3 字幕切分与向量化模块

##### 2.3.1 切分粒度

- 系统应以**固定粒度**（基于 token 窗口）作为最小检索粒度。
- 每个分片应能够与所属用户、所属视频、分片序号建立映射关系。
- chunk_index 表示分片序号，不代表句子。

##### 2.3.2 向量模型

- 系统应使用**阿里云百炼向量模型**生成字幕分片向量。
- 后端项目中应引入百炼相关依赖包。

##### 2.3.3 向量 ID 规则

- 系统写入向量数据库时，必须使用业务自定义 `id`。
- 写入向量数据库时，除 `id` 外，还必须同时写入至少以下 metadata fields：`user_id`、`bvid`，必要时可附加 `title`、`document_type` 等字段。
- 向量 `id` 命名规则应为：`user_id_bvid_chunk_index`
- 示例：`1001_BV1xx411c7mD_12`
- 该规则应确保在全库范围内唯一，避免不同用户、不同视频、不同分片之间出现 ID 冲突。

**命名规范说明**：
- Java 代码中使用驼峰命名：userId, chunkIndex, totalChunks
- 数据库列使用下划线命名：user_id, chunk_index, total_chunks
- 向量 metadata 使用下划线命名：user_id, chunk_index, total_chunks

##### 2.3.4 映射持久化

- 系统在向量写入成功时，必须同步写入 MySQL 映射关系。
- MySQL 中至少应记录以下关联信息：
  - 用户 ID
  - 视频 ID 或业务主键
  - BV 号
  - 分片序号
  - 分片文本
  - 向量 ID
- 映射关系用于删除、排查、重建与审计。

#### 2.4 向量存储模块

##### 2.4.1 向量存储位置

- 系统应使用 DashVector 作为向量存储。
- 应接入本地提供的 `spring-ai-alibaba-starter-dashvector-store` 包。
- 后端应引入 Dash 相关 SDK 依赖。

##### 2.4.2 写入规则

- 系统应基于固定粒度分片逐条或批量写入 DashVector。
- 写入时必须使用业务方生成的向量 `id`，不得依赖 DashVector 自动生成 ID。
- 系统应将 `Document.metadata` 中的业务元数据同步写入 DashVector 的 `fields`，以便后续通过 Spring AI 抽象过滤能力执行按用户、按视频范围的检索。

##### 2.4.3 删除规则

- DashVector 删除操作必须基于 `id` 执行。
- 系统**绝不支持**按过滤条件直接删除向量数据。
- 删除前必须先从 MySQL 中查询该视频对应的全部向量 `id`，再执行批量删除。
- 若数据库中不存在映射关系，则系统不得盲目尝试条件删除。

#### 2.5 视频管理模块

##### 2.5.1 已导入视频列表

- 系统应支持用户查看自己已导入的视频列表。
- 列表至少应展示：
  - 视频标题
  - BV 号
  - 导入时间
  - 分片数量或索引状态

##### 2.5.2 删除视频

- 用户应能够删除自己已导入的单个视频。
- 删除视频时，系统必须按顺序完成以下动作：
  - 校验视频归属用户
  - 查询 MySQL 中该视频的向量 ID 映射
  - 调用 DashVector 按 ID 批量删除向量
  - 若向量删除失败，则整体删除操作失败，业务数据不删除，用户需重试
  - 若向量删除成功，则删除数据库中的视频与分片映射记录
  - 删除与该视频绑定的单视频会话记录

#### 2.6 对话与问答模块

##### 2.6.1 对话创建

- 用户登录后应支持创建新对话。
- 创建新对话时，系统应要求用户选择对话范围：
  - 单视频对话
  - 全部视频对话

##### 2.6.2 单视频对话

- 单视频对话仅允许在指定视频的字幕知识范围内进行检索和回答。
- 检索范围应限制为当前用户指定视频下的分片。
- 实现上应至少基于 `user_id` 与 `bvid` 两个 metadata fields 进行联合过滤。

##### 2.6.3 全部视频对话

- 全部视频对话允许在当前用户已导入的全部视频知识范围内进行检索和回答。
- 检索范围应限制在当前用户的数据空间内，不得跨用户检索。
- 实现上应至少基于 `user_id` metadata field 进行过滤，以避免不同用户在共享向量库中互相检索到对方数据。

##### 2.6.4 多轮对话

- 系统应支持多轮对话。
- 在同一会话中，系统应结合历史消息维持上下文连续性。
- 多轮对话范围不得突破会话创建时选择的知识边界。

##### 2.6.5 回答生成方式

- 系统应采用基础 RAG 流程实现问答。
- 本阶段应使用 `QuestionAnswerAdvisor` 实现检索增强问答。
- 本阶段不要求引入复杂 RAG 机制。

##### 2.6.6 对话模型

- 系统问答模型应使用 **OpenAI 兼容格式**的大模型接口。
- 系统应支持用户或系统配置自定义模型访问 `URL` 与 `API Key`。
- 后端项目中应引入 OpenAI 相关依赖包。

##### 2.6.7 回答内容

- 系统回答应基于已检索到的字幕分片生成。
- 系统应支持前端控制是否返回原文片段。
- 系统至少应支持返回以下信息中的部分或全部：
  - 生成答案
  - 来源视频
  - 命中分片片段
  - 是否返回原文

#### 2.7 会话管理模块

##### 2.7.1 会话持久化

- 系统应保存用户的对话会话及消息记录。
- 会话记录应区分单视频对话与全视频对话。

##### 2.7.2 会话与视频关联

- 单视频对话应与具体视频建立关联关系。
- 全视频对话不绑定单一视频。

##### 2.7.3 删除联动规则

- 删除视频时，应同步删除该视频下关联的单视频对话记录。
- 全视频对话默认保留，但后续检索不得再命中已删除视频的分片数据。
- 若全视频历史消息中存在已删除视频的旧引用，该历史文本可保留为历史记录，不作为新的检索数据来源。

#### 2.8 前端页面需求

- 系统前端应至少包含以下页面：
  - 登录页面
  - 注册页面
  - 对话页面
  - 导入视频页面
  - 已导入视频列表页面
- 在已导入视频列表页面中，应支持对视频执行删除操作。

## 三、非功能需求

### 1. 技术架构要求

- 后端采用 `Spring Boot`。
- AI 应用框架采用 `Spring AI`。
- B 站读取、百炼向量、DashVector 等能力允许使用 `Spring AI Alibaba` 相关组件实现。
- 前端采用 `Vue`。
- 关系型数据存储采用 `MySQL`。

### 2. 性能要求

- 系统应支持个人知识库场景下的视频导入与问答。
- 在数据规模适中的情况下，问答请求应具备可接受的交互响应时间。
- 删除视频时，应支持按映射结果批量删除向量数据，避免逐条低效删除。

### 3. 安全要求

- 系统应保证用户数据隔离。
- 系统应避免将一个用户的视频、向量、会话暴露给其他用户。
- B 站账号凭证信息应在传输与存储过程中受到保护。
- B 站账号凭证不得在后端持久化存储；前端本地保存时默认采用 `sessionStorage`，并尽量缩小暴露范围。
- 日志中不应明文打印完整敏感凭证。

### 4. 一致性要求

- 视频导入、向量写入、映射入库应保证业务一致性。
- 删除视频时，向量删除失败则整体删除操作失败，业务数据不删除，用户需重试。
- 向量删除成功后，数据库记录才会被删除，确保数据一致性。

### 5. 可维护性要求

- 业务模块应清晰拆分为用户管理、视频导入、字幕切分、向量管理、问答会话等模块。
- 向量 ID 规则、分片规则、删除联动规则应集中管理，避免散落在不同模块中。

## 四、数据需求

### 1. 核心数据实体

系统至少应包含以下核心实体：

- 用户
- 视频
- 视频分片
- 向量映射
- 会话
- 会话消息

### 2. 关键关系要求

- 一个用户可导入多个视频。
- 一个视频可拆分为多个字幕分片。
- 一个字幕分片对应一个向量 ID。
- 一个单视频会话关联一个视频。
- 一个全视频会话关联一个用户，不绑定具体单视频。

### 3. 建议字段

建议业务表中至少保留下列字段：

- 用户表：用户 ID、账号、密码摘要、创建时间
- 视频表：视频主键、用户 ID、BV 号、标题、简介、导入时间、状态
- 分片表：分片主键、视频主键、用户 ID、分片序号、分片文本
- 向量映射表：映射主键、用户 ID、视频主键、分片主键、向量 ID
- 会话表：会话主键、用户 ID、会话类型、关联视频 ID、创建时间
- 消息表：消息主键、会话主键、角色类型、消息内容、创建时间

## 五、接口与流程需求

### 1. 典型导入流程

1. 用户登录系统。
2. 用户在导入页面输入 `SESSDATA`、`bili_jct`、`buvid3`。
3. 用户输入 `BV号` 或 `视频URL`。
4. 系统检查该用户下是否已存在相同 `BV号`。
5. 若已存在，则提示用户“请先删除后再导入”。
6. 若不存在，则调用 Bilibili Reader 获取标题、简介、字幕正文等数据。
7. 系统按固定粒度切分字幕（基于 token 窗口）。
8. 系统生成向量 ID。
9. 系统调用百炼向量模型生成 Embedding。
10. 系统写入 DashVector。
11. 系统写入 MySQL 视频、分片、向量映射记录。

### 2. 典型问答流程

1. 用户登录系统。
2. 用户创建新对话。
3. 用户选择“单视频对话”或“全部视频对话”。
4. 系统根据当前会话范围检索当前用户对应的分片数据。
5. 系统通过 `QuestionAnswerAdvisor` 组织检索上下文。
6. 系统调用 OpenAI 兼容模型接口生成回答。
7. 系统返回答案及可选的来源原文信息。

### 3. 典型删除流程

1. 用户在已导入视频列表中选择删除某个视频。
2. 系统校验视频归属。
3. 系统查询该视频关联的全部向量 ID。
4. 系统调用 DashVector 批量删除向量。
5. 若向量删除失败，则整体删除操作失败，业务数据不删除，用户需重试。
6. 若向量删除成功，则删除视频、分片、映射记录。
7. 删除该视频关联的单视频会话与消息。
8. 系统返回删除结果。

## 六、约束条件

### 1. 技术约束

- RAG 流程以简单实现为主，不引入复杂编排。
- 核心问答实现采用 `QuestionAnswerAdvisor`。
- 向量模型与对话模型分离：
  - 向量模型使用阿里云百炼
  - 对话模型使用 OpenAI 兼容接口

### 2. 数据约束

- 向量数据删除必须依赖本地数据库映射，不得尝试条件删除。
- 向量 `id` 必须由业务系统自行构造并保证唯一性。
- 问答数据源以字幕为主，不保证覆盖视频中的所有语义信息。

### 3. 业务约束

- 系统仅面向登录用户开放。
- 系统仅支持单视频导入，不支持批量导入。
- 系统当前仅支持有字幕的视频知识化处理。

## 七、验收标准

- 用户可完成注册、登录，并且不同用户之间的数据彼此隔离。
- 用户输入 `SESSDATA`、`bili_jct`、`buvid3` 以及 `BV号` 或 `视频URL` 后，系统可成功导入带字幕的视频。
- 对已导入视频，系统可完成固定粒度切分、向量生成、DashVector 写入及 MySQL 映射保存。
- 同一用户重复导入相同 `BV号` 时，系统提示已存在，要求先删除后再导入。
- 用户可选择单视频范围或全部视频范围发起多轮问答。
- 问答链路基于 `QuestionAnswerAdvisor` 正常工作，并返回基于字幕内容生成的结果。
- 用户删除视频时，系统可根据 MySQL 中记录的向量 ID 批量删除 DashVector 向量。
- 用户删除视频时，系统同步删除视频记录、分片映射及该视频下的单视频会话。

## 八、后续可扩展项

- 支持更多 B 站导入入口，如收藏夹、UP 主页、投稿列表。
- 支持字幕缺失时接入语音识别转写。
- 支持更复杂的 RAG 流程，如重排、查询改写、摘要缓存和多路检索。
- 支持视频时间戳、原文定位、上下文片段回溯。
- 支持后台管理、向量索引监控与任务补偿机制。
