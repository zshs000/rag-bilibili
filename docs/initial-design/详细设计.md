### 2. GlobalExceptionHandler 全局异常处理器

```java
package com.example.ragbilibili.exception;

import com.example.ragbilibili.common.Result;
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;
import org.springframework.validation.BindException;
import org.springframework.web.bind.MethodArgumentNotValidException;
import org.springframework.web.bind.annotation.ExceptionHandler;
import org.springframework.web.bind.annotation.RestControllerAdvice;

/**
 * 全局异常处理器
 */
@RestControllerAdvice
public class GlobalExceptionHandler {
    private static final Logger logger = LoggerFactory.getLogger(GlobalExceptionHandler.class);

    /**
     * 处理业务异常
     */
    @ExceptionHandler(BusinessException.class)
    public Result<?> handleBusinessException(BusinessException e) {
        logger.warn("业务异常: code={}, message={}", e.getCode(), e.getMessage());
        return Result.error(e.getCode(), e.getMessage());
    }

    /**
     * 处理参数校验异常
     */
    @ExceptionHandler({MethodArgumentNotValidException.class, BindException.class})
    public Result<?> handleValidationException(Exception e) {
        String message = "参数校验失败";
        if (e instanceof MethodArgumentNotValidException) {
            MethodArgumentNotValidException ex = (MethodArgumentNotValidException) e;
            if (ex.getBindingResult().hasErrors()) {
                message = ex.getBindingResult().getAllErrors().get(0).getDefaultMessage();
            }
        } else if (e instanceof BindException) {
            BindException ex = (BindException) e;
            if (ex.getBindingResult().hasErrors()) {
                message = ex.getBindingResult().getAllErrors().get(0).getDefaultMessage();
            }
        }
        logger.warn("参数校验异常: {}", message);
        return Result.error(ErrorCode.PARAM_ERROR.getCode(), message);
    }

    /**
     * 处理系统异常
     */
    @ExceptionHandler(Exception.class)
    public Result<?> handleException(Exception e) {
        logger.error("系统异常", e);
        return Result.error(ErrorCode.SYSTEM_ERROR.getCode(), "系统错误，请联系管理员");
    }
}
```

### 1. BVIDParser BV号解析工具

```java
package com.example.ragbilibili.util;

import com.example.ragbilibili.exception.BusinessException;
import com.example.ragbilibili.exception.ErrorCode;

import java.util.regex.Matcher;
import java.util.regex.Pattern;

/**
 * BV号解析工具类
 */
public class BVIDParser {
    /**
     * BV号正则表达式
     * 匹配格式：BV + 10位字符（数字和大小写字母）
     */
    private static final Pattern BVID_PATTERN = Pattern.compile("(BV[a-zA-Z0-9]{10})");

    /**
     * 从输入字符串中解析 BV 号
     *
     * @param input BV号或包含BV号的URL
     * @return BV号
     * @throws BusinessException 解析失败时抛出
     */
    public static String parse(String input) {
        if (input == null || input.trim().isEmpty()) {
            throw new BusinessException(ErrorCode.BVID_PARSE_ERROR);
        }

        Matcher matcher = BVID_PATTERN.matcher(input);
        if (matcher.find()) {
            return matcher.group(1);
        }

        throw new BusinessException(ErrorCode.BVID_PARSE_ERROR);
    }
}
```

package com.example.ragbilibili.util;

/**
 * 向量ID生成工具类
 *
 * 向量ID格式：user_id_bvid_chunk_index

### 3. PasswordEncoder 密码加密工具

```java
package com.example.ragbilibili.util;

import org.mindrot.jbcrypt.BCrypt;

/**
 * 密码加密工具类
 * 使用 jBCrypt 库实现 BCrypt 加密
 * 依赖：org.mindrot:jbcrypt:0.4
 */
public class PasswordEncoder {
    /**
     * 加密密码
     *
     * @param rawPassword 原始密码
     * @return 加密后的密码
     */
    public static String encode(String rawPassword) {
        return BCrypt.hashpw(rawPassword, BCrypt.gensalt());
    }

    /**
     * 验证密码
     *
     * @param rawPassword 原始密码
     * @param encodedPassword 加密后的密码
     * @return 是否匹配
     */
    public static boolean matches(String rawPassword, String encodedPassword) {
        return BCrypt.checkpw(rawPassword, encodedPassword);
    }
}
```

**依赖说明**：
- 使用 jBCrypt 库（`org.mindrot:jbcrypt:0.4`）实现 BCrypt 加密
- 不依赖 Spring Security，是一个独立的 BCrypt 实现
- Maven 依赖：
  ```xml
  <dependency>
      <groupId>org.mindrot</groupId>
      <artifactId>jbcrypt</artifactId>
      <version>0.4</version>
  </dependency>
  ```

     */
    VideoResponse importVideo(ImportVideoRequest request, Long userId);

    /**
     * 获取用户视频列表
     *
     * @param userId 用户ID
     * @return 视频列表
     */
    List<VideoResponse> listVideos(Long userId);

    /**
     * 获取视频详情
     *
     * @param videoId 视频ID
     * @param userId 用户ID
     * @return 视频响应
     */
    VideoResponse getVideo(Long videoId, Long userId);

    /**
     * 删除视频（级联删除分片、向量、会话）
     *
     * @param videoId 视频ID
     * @param userId 用户ID
     */
    void deleteVideo(Long videoId, Long userId);

```

**VideoService 实现说明**：

#### 1. importVideo 方法实现流程

```
1. 解析 BV 号（如果输入是 URL）
2. 检查该用户下是否已存在相同 BV 号的视频
3. 创建 video 记录，状态设置为 IMPORTING
4. 调用 Bilibili Reader 获取视频数据
5. 筛选 document_type=content 的文档
6. 使用 TokenTextSplitter 切分文本
7. 为每个分片补充 metadata 和自定义 id
8. 调用百炼向量模型生成 Embedding
9. 写入 DashVector
10. 写入 MySQL (chunk, vector_mapping)
11. 更新 video 状态为 SUCCESS
12. 统计 chunk 数量并返回
```

**状态管理规则**：
- **IMPORTING**：创建 video 记录时立即设置，表示导入正在进行中
- **SUCCESS**：所有步骤成功完成后设置
- **FAILED**：任何步骤失败时设置

**失败处理规则**：
- 如果在步骤 1-3 失败：不需要清理，直接抛异常
- 如果在步骤 4-8 失败：更新 video 状态为 FAILED，不删除 video 记录（用户可查看失败原因）
- 如果在步骤 9（写入 DashVector）失败：
  - 更新 video 状态为 FAILED
  - 不需要清理向量（因为写入失败）
  - 不写入 chunk 和 vector_mapping
- 如果在步骤 10（写入 MySQL）失败：
  - 更新 video 状态为 FAILED
  - **需要清理已写入的向量**（调用 DashVector 删除）
  - 这是 V1 阶段的已知风险点（分布式事务问题）

**用户可见性**：
- IMPORTING 状态的视频：用户可以在列表中看到，但不能创建会话或对话
- SUCCESS 状态的视频：用户可以正常使用
- FAILED 状态的视频：用户可以在列表中看到，显示失败原因，可以删除后重新导入

**chunkCount 计算**：
- 在导入成功后，通过查询 chunk 表统计该视频的分片数量
- SQL: `SELECT COUNT(*) FROM chunk WHERE video_id = ?`
- 返回给前端时包含在 VideoResponse 中

#### 2. deleteVideo 方法实现流程

```
1. 校验视频归属（确保 video.user_id = userId）
2. 查询 vector_mapping 表获取所有向量 ID
3. 调用 DashVector 批量删除向量（失败则抛异常，整体回滚）
4. 查询该视频关联的单视频会话 ID
5. 删除对应会话下的 message 记录
6. 删除 session 表中 video_id = videoId 的单视频会话
7. 删除 vector_mapping 表记录
8. 删除 chunk 表记录
9. 删除 video 表记录
```

**全视频对话的处理**：
- 全视频对话（session_type = ALL_VIDEOS）不会被删除
- 历史消息保留，但 V1 不做“来源视频已删除”标记
- 后续检索时，由于向量已删除，不会再命中该视频的内容
- V1 不追踪消息级来源视频关系，历史消息仅作为已生成内容保存

package com.example.ragbilibili.service;

import org.springframework.web.servlet.mvc.method.annotation.SseEmitter;

/**
 * 对话服务接口
 */
public interface ChatService {
    /**
     * 流式发送消息
     *
     * @param sessionId 会话ID
     * @param content 消息内容
     * @param userId 用户ID
     * @return SSE Emitter
     */

}
```

**流式响应实现说明**：

ChatService 的 `streamMessage` 方法应按以下流程实现：

1. **保存用户消息**：立即将用户消息保存到数据库，获得 `userMessageId`
2. **发送 start 事件**：通过 SSE 发送开始事件
   ```json
   {"type": "start", "userMessageId": 1}
   ```
3. **执行 RAG 检索**：根据会话类型构建过滤条件，检索相关分片
4. **流式生成回答**：调用 Spring AI 的 `ChatClient.stream()` 方法
5. **发送 content 事件**：每收到一个增量片段，立即通过 SSE 发送
   ```json
   {"type": "content", "delta": "这个"}
   ```
6. **保存助手消息**：流式生成完成后，将完整内容保存到数据库，获得 `assistantMessageId`
7. **发送 end 事件**：通过 SSE 发送结束事件
   ```json
   {"type": "end", "userMessageId": 1, "assistantMessageId": 2, "fullContent": "这个视频主要讲解了..."}
   ```
8. **关闭连接**：完成 SSE 连接

**关键点**：
- `userMessageId` 在 start 和 end 事件中都返回，用于前端关联一次完整的问答交互
- `assistantMessageId` 仅在 end 事件中返回，用于前端存储助手消息的数据库 ID
- `fullContent` 用于前端校验或重新渲染完整内容
- 异常处理：如果流式生成过程中出现异常，应通过 SSE 发送 error 事件并关闭连接

## 十三、事务管理设计

### 1. 事务管理原则

- 禁止使用分布式事务（V1 阶段不引入 Seata 等分布式事务框架）
- 采用最终一致性策略，通过补偿机制保证数据最终一致
- 先写向量，后写数据库
- 删除操作使用本地事务

**V1 阶段的简化处理**：
- 当前删除流程存在分布式事务隐患：DashVector 删除成功但数据库删除失败时，无法回滚向量删除
- V1 阶段暂不引入分布式事务框架，采用"向量删除失败即整体失败"的策略
- 后续版本可考虑引入补偿机制或分布式事务框架

### 2. 视频删除事务设计

```java
@Override
@Transactional(rollbackFor = Exception.class)
public void deleteVideo(Long videoId, Long userId) {
    // 1. 校验视频归属
    Video video = videoMapper.selectById(videoId);
    if (video == null || !video.getUserId().equals(userId)) {
        throw new BusinessException(ErrorCode.VIDEO_NOT_FOUND);
    }

    // 2. 查询向量 ID 列表（事务内）
    List<String> vectorIds = vectorMappingMapper.selectVectorIdsByVideoId(videoId);

    // 3. 删除 DashVector 向量（失败即抛出异常，事务回滚）
    if (!vectorIds.isEmpty()) {
        vectorStore.delete(vectorIds);  // 失败会抛异常，事务回滚
    }

    // 4. 查询该视频关联的单视频会话 ID
    List<Long> sessionIds = sessionMapper.selectIdsByVideoId(videoId);

    // 5. 先删消息，再删会话
    if (!sessionIds.isEmpty()) {
        messageMapper.deleteBySessionIds(sessionIds);
    }
    sessionMapper.deleteByVideoId(videoId);

    // 6. 删除向量映射、分片与视频记录
    vectorMappingMapper.deleteByVideoId(videoId);
    chunkMapper.deleteByVideoId(videoId);
    videoMapper.deleteById(videoId);
}
```

**说明**：
- 向量删除失败时，`vectorStore.delete()` 会抛出异常
- 事务回滚，数据库记录不删除
- 用户需要重试整个删除操作
- 不使用 try-catch 吞异常，确保数据一致性

**V1 阶段的已知问题**：
- 存在分布式事务隐患：如果 DashVector 删除成功但数据库删除失败，事务回滚无法撤销已删除的向量
- 这种情况下会出现"向量已删除但数据库记录还在"的不一致状态
- V1 阶段暂不处理此场景，后续版本可引入补偿机制或分布式事务框架（如 Seata）
- 实际场景中，数据库删除失败的概率较低，V1 阶段可接受此风险
