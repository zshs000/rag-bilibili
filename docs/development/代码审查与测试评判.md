# 代码审查与测试评判

本文档给 AI agent 使用。做代码审查、尤其在准备说"测试覆盖不够""测试漏了 XX"之前，按本文档判断。

## 什么时候使用

当任务满足任一条件时，使用本指南：

- 用户要求"做代码审查""review 一下""挑问题"
- 准备批评测试用例写得不够、断言不够、覆盖不全
- 准备建议新增测试用例
- 评判一份 PR/diff 的质量

以下任务不使用本指南：

- 用户只要"读懂代码"、"解释功能"
- 用户在写新代码，让你跟着做 TDD（这种场景用 TDD 自己的节奏即可，不需要审查纪律）
- 纯文档审查

## 前置背景：项目按 TDD 写代码

**本项目大体按 TDD 节奏写代码：先 Red 再 Green，每一轮 Red 只对应一条当前要实现的行为。**

这条背景直接决定了审查时的判断尺度：

- TDD 不要求一上来就把所有边界、所有 null 安全、所有空集合分支都写齐。
- TDD 要求的是：**每一条被实现的行为都有一个 Red 测试驱动它出生**。
- 所以"边界覆盖不全"在 TDD 上下文里**不必然是 bug**——它可能只是还没遇到那个 failure。
- 但"已经存在的 `if`/已经做出的核心契约承诺**没有任何测试解释它**"在 TDD 上下文里就是 bug——这个分支或契约没有出生证。

不理解这条背景，会把 TDD 的阶段性留白当成质量问题，写出一堆"以后改坏了测不到"的伪批评。

## 评判原则

测试是**当前规约的可执行表达**，不是**防御未来错误的万能护栏**。

- 测试是抽样，不是证明。它本质上不可能防御一切未来的错误代码。
- 审查时不要用"如果有人以后把代码写反了，测试也不报警"作为论据——这是测试万能论，任何测试都顶不住这种攻击。
- 审查时要问的是：**这条规约/这个分支，在当前代码里有没有一个测试在替它说话？**

```text
错误的批评：
"VideoServiceImpl.deleteVideo 里如果以后有人把 dashVectorStore.delete 写在
videoDeleteTxService.deleteVideoData 之前，现在的测试也照样通过。"

正确的批评：
"本次重构的核心承诺就是'先 DB 后 DashVector'，而测试里没有任何
InOrder 断言显式表达这条契约。这条契约目前只活在文档里，不活在测试里。"
```

差别在于：前者是"未来防御"，后者是"当前规约缺失"。

## 该加测试的两种情形

只有满足以下任一条件时，才应该建议新增测试：

### 1. 本次改动的核心契约/不变量

如果改动的灵魂就在某条不变量上（跨服务调用顺序、用户感知行为、数据一致性方向），那条不变量值得一个直接表达它的测试。

```text
例：本次"先删 DB 后删 DashVector"的重构，整个 PR 存在的理由就是这个顺序。
   理应有一条 InOrder 测试作为它在测试层的可执行规约。
```

判断方法：读这次 commit/PR 的描述或设计文档，看它承诺的核心不变量是什么；
然后去测试里找有没有用例**直接断言**这条不变量。找不到，就是真问题。

### 2. 生产代码里已经存在的 `if`/分支，但没有用例解释它

如果你在 service/controller 里看到一个 `if`，但测试里**没有任何用例覆盖这个分支**，
说明这个 `if` 在 TDD 节奏下没有出生证——要么作者不是真 TDD，要么这个 `if` 本来就多余。

```java
// 生产代码：
if (!sessionIds.isEmpty()) {
    messageMapper.deleteBySessionIds(sessionIds);
}
```

```text
该问的问题：测试里有没有 "sessionIds 为空" 的用例？
如果没有，这个 if 就是凭空冒出来的——要么补一条 Red，要么说明这个 if 多余可以删。
```

## 不该提的两种情形（避免过度审查）

### 1. null、空集合等边界穷举

按 TDD 节奏可以等遇到具体 failure 再补。

```text
反例（不应该作为批评提出）：
"vectorMappingMapper.selectVectorIdsByVideoId 如果返回 null，
VideoServiceImpl 的 vectorIds.isEmpty() 会 NPE，测试没覆盖这个。"
```

这种边界穷举式批评在 TDD 上下文里是过头的。MyBatis selectList 默认返回空 list，不必第一轮就替它写防御性测试。如果将来真出现 null 导致 NPE 的 incident，那时再补一条 Red。

注意区分：如果**生产代码里已经写了 null 检查**（如 `Objects.requireNonNull`），那这个分支就回到第 2 种情形，应该有测试解释它。

### 2. 函数内部自然 dataflow

如果两个调用的顺序是被自然的 dataflow 决定的（A 不在前面，B 就拿不到数据），就不需要 `InOrder` 反复 assert。代码长得对就够了。

```text
反例（属 over-test）：
"VideoDeleteTxService 里 vectorMappingMapper.selectVectorIdsByVideoId 必须在
vectorMappingMapper.deleteByVideoId 之前调用，测试没有 InOrder 验证这一对。"
```

这条顺序不是契约，是 dataflow——查询不在前面就没东西可返回。这种顺序代码读一眼就知道，不需要测试反复 assert。

跟"先删 DB 后删 DashVector"的区别：后者是**跨服务的独立契约**，源码结构上看不出"必须先后"，必须由测试规约住；前者是**函数内部数据依赖**，源码本身就强制了顺序。

## 测试反模式（这些是真 bug）

跟 TDD 节奏无关，看到就该提。

### 1. mock 注入设定下永远成立的 verify（同义反复）

```java
@Mock VideoDeleteTxService videoDeleteTxService;
@Mock VectorMappingMapper vectorMappingMapper;
@InjectMocks VideoServiceImpl videoService;

@Test
void test() {
    when(videoDeleteTxService.deleteVideoData(...)).thenReturn(vectorIds);
    videoService.deleteVideo(...);

    verify(vectorMappingMapper, never()).deleteByVideoId(any());  // 永远成立
}
```

`videoDeleteTxService` 自己就是 mock，它内部不会触发真实的 `vectorMappingMapper`。这条 `verify(never())` 不论代码怎么写都成立，**它什么都没保护**。

### 2. 测试名喊出契约，但断言没真正覆盖契约

```java
@Test
void deleteVideoShouldDeleteVectorsAfterDatabaseDelete() {
    // ...
    verify(videoDeleteTxService, times(1)).deleteVideoData(100L, 1L);
    verify(dashVectorStore, times(1)).delete(vectorIds);
    // 没有 InOrder！"after" 这个词在断言里没有落地。
}
```

测试名喊了"after database delete"，但断言里只验了两个调用各发生 1 次，没有任何顺序断言。把代码反过来写（DashVector 在前），这个测试照样绿。**测试名在说谎。**

## 多层防御：测试不是唯一手段

正确性是多层防御的合力。审查时优先看更硬的层能不能挡住错误，不要把所有压力堆到测试上。

防御从硬到软：

1. **架构约束**：让错误代码写不出来。
2. **类型/契约**：让错误代码编译不过。
3. **测试**：让错误代码跑不过。
4. **可观测性**：让错误代码上线了能被发现。
5. **文档**：让人读得懂为什么这样写。

举例：

```text
本次重构后，VideoServiceImpl 里 vectorMappingMapper / sessionMapper / messageMapper
三个 @Autowired 字段已经没人用了。

如果只加测试不删字段：后人仍然有空间在 deleteVideo 里重新混进 mapper 调用，
绕过 VideoDeleteTxService，把"先 DB 后 DashVector"的顺序破坏掉。
测试再多也防不住这种结构性退化。

如果一并删字段：后人想在 deleteVideo 里直接调 mapper，会被编译期挡住——
他要么去注新的 mapper（会引起 review 注意），要么必须走 VideoDeleteTxService。
这是比加测试更硬的防御。
```

审查时如果发现"删字段就能挡住的错误，作者用测试在挡"，应该建议先做结构清理。

```text
本次 DashVector 删除失败被吞掉，文档承认"用户不感知，留下孤儿向量"。
这种'线上才会出现的失真'不是测试能管的——单测里 mock 永远是确定的成功或失败。
应该用 micrometer counter / 告警 / 补偿任务 来管，不要指望测试。
```

审查时如果发现"线上失真问题作者打算用测试覆盖"，应该建议挪到监控/补偿层。

## 审查输出格式

审查发现的问题按以下格式输出，避免堆罗列：

```text
严重程度（高/中/低/残余风险）
  文件:行号
  问题描述（一句话说清楚）
  论据（为什么这是问题，引用上面的判断原则）
```

严重程度的判断尺度：

- **高**：本次改动的核心契约缺失测试规约（情形 1）、测试反模式
- **中**：已存在的 `if` 没有出生证（情形 2）、结构清理不彻底导致防御层退化
- **低**：可观测性退步、日志措辞可能误导、注释/文档与代码轻微不一致
- **残余风险**：当前不算 bug 但值得标注，如重复操作的孤儿数据问题

## 完整反面教材：本次 `VideoServiceImpl.deleteVideo` 重构的审查

记录一次审查者自己踩坑的过程，作为后人对照。

**改动**：把 `deleteVideo` 从"先 DashVector、后 DB"改成"先 DB、后 DashVector"，DashVector 失败用户不感知。删除逻辑搬到 `VideoDeleteTxService`。

**审查者第一轮（包含过头批评）**：

- ✗ "VideoDeleteTxServiceTest 没有 InOrder 验证 selectVectorIdsByVideoId 在 deleteByVideoId 之前调用"
  → 这是函数内部 dataflow，属 over-test。不该提。
- ✗ "没有测试 vectorIds 为 null 的 NPE 场景"
  → 边界穷举，TDD 节奏可以等。不该提。
- ✗ "如果以后有人把 dashVectorStore.delete 写到 deleteVideoData 之前，测试不会报"
  → 测试万能论，论据错。该改成"核心契约缺失测试规约"。

**审查者第一轮（站得住的批评）**：

- ✓ "本次重构的核心承诺是'先 DB 后 DashVector'，但 `VideoServiceImpl` 的测试里没有 `InOrder` 断言显式表达这条契约" → 情形 1
- ✓ "`if (!sessionIds.isEmpty())` 在 service 里已经写出来，但测试里没有'sessionIds 为空'的用例" → 情形 2
- ✓ "`verify(vectorMappingMapper, never()).deleteByVideoId(any())` 在 mock 注入下永远成立" → 反模式 1
- ✓ "测试名叫 `deleteVideoShouldDeleteVectorsAfterDatabaseDelete` 但断言里没有 InOrder" → 反模式 2
- ✓ "`vectorMappingMapper` / `sessionMapper` / `messageMapper` 三个 @Autowired 字段已成死字段，应一并清掉" → 多层防御：架构清理优先于测试
- ✓ "DashVector 失败被吞，没有 counter / 告警 / 补偿接口" → 多层防御：线上失真用可观测性管

**经验**：

- 审查时容易在"凑数量"的诱惑下把边界穷举、dataflow over-test、未来防御都列出来，显得审查很深入。
- 实际上这些过头批评会污染真问题，让被审查者抓不到重点，也让审查纪律本身被怀疑。
- 宁可少而准，不要多而散。

## 禁止事项

- 禁止用"以后有人改坏了测试不报"作为论据。
- 禁止建议为函数内部 dataflow 加 `InOrder` 断言。
- 禁止在 TDD 节奏下批评 null/空集合等边界穷举的缺失。
- 禁止建议"加测试"来挡架构/类型层就能挡住的错误。
- 禁止建议"加测试"来覆盖线上才会出现的失真。
- 禁止在审查输出里不分严重程度地一锅端，必须分级。
