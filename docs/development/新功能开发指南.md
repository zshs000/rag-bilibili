# 新功能开发指南

本文档给 AI agent 使用。执行新功能任务时，按下面规则做。

## 什么时候使用

当任务满足任一条件时，使用本指南：

- 新增后端接口
- 修改接口路径、参数、响应字段、错误码
- 修改 SSE 事件名、事件顺序或 payload
- 一个功能需要前端和后端共同改动
- 用户明确说“开发新功能”“加一个功能”“前后端一起做”

以下任务不使用本指南：

- 不改接口的 bugfix
- 纯后端内部重构
- 纯前端样式或交互调整
- 测试补充
- 文档错别字修正

## 总规则

接口变更时，先定契约，再写代码。

优先级：

1. 当前代码和测试用于判断现有行为。
2. `docs/API.md` 用作前后端协作契约。
3. 如果代码和 `docs/API.md` 冲突，先指出冲突，不要猜。

## 执行流程

### 1. 判断是否改接口

先回答这个问题：

```text
这个任务是否改变前后端接口契约？
```

如果答案是“否”，停止使用本指南，按对应目录 `AGENTS.md` 开发。

如果答案是“是”，继续下一步。

### 2. 先改 `docs/API.md`

在写后端或前端代码前，先更新 `docs/API.md`。

每个接口必须写清：

- 请求方法
- 请求路径
- 是否需要登录
- path 参数
- query 参数
- body 结构
- 成功响应 `data` 结构
- 业务错误码
- SSE 事件名、顺序、payload，如果是流式接口
- 兼容性影响，如果会破坏旧调用方

不要写空话。另一个 agent 只看 `docs/API.md` 时，必须能完成对接。

### 3. 确认实现范围

根据用户当前窗口的目标，只做一端。

后端窗口只做：

- 后端代码
- 后端测试
- 必要的 `docs/API.md` 修正

前端窗口只做：

- 前端代码
- 必要的 `docs/API.md` 修正

如果用户要求当前窗口同时做前后端，先提示风险：

```text
当前窗口同时改前后端会扩大上下文和改动范围，容易漏同步 API 契约，也更难并行验证。
建议先更新 docs/API.md，再按后端和前端两个阶段实现。
如果你确认要在当前窗口一起做，我会继续，但会先完成接口契约。
```

用户确认后，可以在当前窗口同时做前后端，但必须先完成 `docs/API.md`。

### 4. 后端实现要求

后端实现时必须遵守：

- Controller 返回 `Result<T>`
- 业务失败抛 `BusinessException` + `ErrorCode`
- 不手写 JSON 响应
- 不跨层调用，保持 controller -> service -> mapper
- 认证使用 Sa-Token，保持 `Authorization: Bearer <token>`
- 涉及用户数据时校验当前用户归属

完成后至少说明：

```text
已实现接口：
- METHOD /path

新增或修改错误码：
- CODE: 含义

验证：
- 执行了什么命令
- 结果如何
```

### 5. 前端实现要求

前端实现时必须遵守：

- API 调用写在 `src/api/`
- REST 请求使用现有 Axios 封装
- SSE 使用现有流式 helper
- 按 `Result<T>` 读取响应：`{ code, message, data }`
- 不硬编码后端地址
- 登录态和 `1004` / `NOT_LOGGED_IN` 行为保持一致

完成后至少说明：

```text
已对接接口：
- METHOD /path

页面入口：
- 路由或组件

验证：
- 执行了什么命令
- 结果如何
```

## 多窗口并行提示

如果用户要开两个 AI 窗口并行开发，先更新 `docs/API.md`，然后把下面内容发给用户：

```text
docs/API.md 已作为本功能的接口契约更新。

窗口 1：打开 rag-bilibili-server/，实现 docs/API.md 中定义的后端接口。
窗口 2：打开 rag-bilibili-front/，对接 docs/API.md 中定义的前端功能。

两个窗口都要遵守各自目录下的 AGENTS.md。
如果发现 docs/API.md 与代码冲突，先说明冲突，不要自行猜测。
```

## 冲突处理

如果实现时发现 `docs/API.md` 和代码不一致：

1. 先停止基于猜测继续写。
2. 明确列出冲突：

```text
冲突：
- docs/API.md 写的是：...
- 当前代码实际是：...
```

3. 如果能判断文档过期，更新 `docs/API.md`。
4. 如果会改变已有行为，先问用户。

## 禁止事项

- 禁止后端改了响应字段但不更新 `docs/API.md`。
- 禁止前端靠猜测字段名开发。
- 禁止两个窗口各自定义一套接口。
- 禁止为了小 bugfix 强行改 API 文档。
- 禁止把历史设计文档当成当前接口事实。

## 最低验证

后端：

```bash
cd rag-bilibili-server
mvn test
```

前端：

```bash
cd rag-bilibili-front
npm run build
```

如果因为数据库、环境变量、模型服务或依赖缺失无法验证，必须在最终回复中说明。
